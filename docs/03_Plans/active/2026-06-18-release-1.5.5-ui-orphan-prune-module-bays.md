# Release 1.5.5 — UI for Orphan Prune + Module-Bay Readiness

## Goal

Make two operator workflows self-serve in the NetBox UI, with no shell or CSV:

1. **Scope reconciliation / orphan prune** — see in-scope vs orphan device counts
   on the sync page and delete the orphans with a button.
2. **Module-bay readiness** — see how many module bays are missing and create them
   directly in NetBox with a button (replacing the generate-CSV → import round
   trip).

## Constraints

- No new background-job infrastructure or model migrations: the live Forward
  query is fast, so the read-only "check" views run synchronously (mirroring the
  existing dependency-preview view), and the mutating actions are synchronous
  POSTs (mirroring the run/validate views).
- Single source of truth: the UI and the CLI commands share the same compute /
  prune / create-bays utilities.
- Mutating actions are permission-gated (`dcim.delete_device`,
  `dcim.add_modulebay`) and confirm before acting.
- Orphan prune keeps the empty-Forward-result safety guard; module-bay create is
  idempotent and skips bays whose device is not yet in NetBox.

## Touched Surfaces

- `forward_netbox/utilities/scope_reconciliation.py` — NEW; `compute_scope_
  reconciliation` + `prune_orphan_devices` (factored out of the audit command,
  which now delegates).
- `forward_netbox/utilities/module_readiness.py` — `fetch_module_rows_for_sync`,
  `compute_module_readiness_for_sync`, `create_missing_module_bays`.
- `forward_netbox/views.py` — four `BaseObjectView`s (scope reconciliation +
  prune, module readiness + create bays) + URLs in the detail view context.
- `forward_netbox/templates/forward_netbox/forwardsync_scope_reconciliation.html`,
  `forwardsync_module_readiness.html` — NEW panels; buttons added to
  `forwardsync.html`.
- `forward_netbox/tests/test_scope_module_ui.py` — NEW; utility + view smoke
  tests.
- `docs/01_User_Guide/troubleshooting.md`, `usage.md` — UI-first guidance.
- `pyproject.toml`, `forward_netbox/__init__.py`, the three release tables —
  bump to 1.5.5.

## Approach

The detail view exposes two new action URLs. Each "check" view runs the live
compute synchronously and renders a panel (counts + sample) with a CSRF POST
button. The POST views resolve the sync, run the shared utility (prune / create),
flash a result message, and redirect. Module bays are created with a plain
`.save()` out of band — MPTT bays only fail to create inside a Branching merge,
not in a direct write.

## Validation

- `forward_netbox.tests.test_scope_module_ui` (readiness compute + idempotent
  create; scope-reconciliation GET + prune POST; module-readiness GET + create
  POST).
- `forward_netbox.tests.test_device_scope_reconciliation_audit_command` (command
  still green after delegating to the shared util).
- Full suite; local CI mirror; GitHub CI on both matrices.

## Rollback

Revert the views, templates, utilities, tests, and version bump. The CLI command
keeps working (it would re-inline its logic on revert). No schema/data impact.

## Decision Log

- Synchronous views (no jobs) because the Forward scope/module queries return in
  seconds and the dependency-preview view already establishes the synchronous-
  live-compute pattern; adding a job + result store would be more surface for no
  real latency benefit.
- "Create bays directly" over "download CSV" per the operator's no-CLI/no-CSV
  workflow; idempotent and device-guarded so re-runs are safe.
- Shared utilities so CLI and UI never diverge; the audit command now delegates.
- `scope_reconciliation.py` uses a comment header (not a module docstring) and no
  `from __future__` import, because a docstring + future import makes
  reorder-python-imports and black oscillate over the blank line and CI (fresh
  hook installs) never converges.
