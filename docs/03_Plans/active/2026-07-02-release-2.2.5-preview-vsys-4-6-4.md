# Release 2.2.5 — operator-selectable Sync Device Tags, dependency-preview fix, vsys job pile-up guard, NetBox 4.6.4

**Date:** 2026-07-02

## Goal
One field feature plus two field bugs surfaced by 2.2.4's error visibility, plus
bump the tested/required NetBox to 4.6.4.

1. **Feature — operator-selectable Sync Device Tags.** The `Forward Device Feature
   Tags` map hardcoded `Prot_BGP`. Blake asked to sync the `Mgmt_*` tags instead.
   Rather than swap one hardcode for another, make it a Source-level multi-select:
   the operator picks exactly which Forward device tags become NetBox device tags.
   (BGP tagging belongs in the routing plugin, not this hardcode.)
2. Dependency preview errored `'ForwardModelResult' object has no attribute 'get'`
   — `_dependency_model_result_summary` called `result.get(...)` but
   `fetcher.model_results` are `ForwardModelResult` dataclasses. (Born broken; the
   preview never worked — 2.2.4 just made the hidden error visible.)
3. "link vsys/vdom parents" job hung in pending — the default-on overlay is
   enqueued after EVERY sync; a slow/large run (full-network fetch + ~782 writes)
   is still active when the next sync fires, so duplicates pile up in PENDING.

## Constraints
- Minor-feature scope: one additive Source parameter + two fixes + a version bump.
  No NetBox schema/migration change.
- The tag query is org-published (query_id-bound); its signature changes, so the
  org copy must be republished (or Blake must Refresh Query IDs) at release.
- Bump min NetBox to 4.6.4 (we test what we require).

## Touched Surfaces
- `forward_netbox/queries/forward_device_feature_tags.nqe` — rewritten from the
  hardcoded `Prot_BGP` filter to a param-driven map:
  `f(forward_netbox_shard_keys, sync_device_tags)` emitting one row per
  device/selected-tag with a derived slug + neutral colour.
- `forward_netbox/utilities/query_registry.py` — `_default_query_parameters` seeds
  `sync_device_tags: []` for any query whose source references it (mirrors the
  shard-keys rule), so the spec declares the parameter and the injection fires.
- `forward_netbox/utilities/query_fetch_execution.py` — `ForwardQueryContext`
  gains a `sync_device_tags` field (resolved from `source.parameters`);
  `_apply_context_tag_parameters` injects the sorted selection into any query that
  declares the parameter.
- `forward_netbox/forms.py` — `sync_device_tags` multi-select on the Source form
  (reuses the `available-tags` API widget; persisted in both `clean()` and
  `save()` parameter dicts).
- `forward_netbox/utilities/model_validation.py` — `clean_forward_source`
  allowlists `sync_device_tags` and type-checks it as a list of strings.
- `forward_netbox/views.py` — `_dependency_model_result_summary` normalizes via
  `result.as_dict()` (tolerates a dict), derives `estimated_changes` = row+delete.
- `forward_netbox/jobs.py` — `_sync_has_active_job` guard; all four post-sync
  overlay enqueues (device-analysis, backfilled, vsys, auto-prune) skip when an
  active (PENDING/RUNNING) job of the same name already exists for the sync.
- `development/.env`, `.github/workflows/ci.yml`, `forward_netbox/__init__.py`
  (`min_version`) — NetBox 4.6.3 → 4.6.4.
- Docs: `README.md`, `docs/README.md`, `docs/01_User_Guide/README.md` (compat rows
  + requirement line), `docs/01_User_Guide/configuration.md` (feature-tags map now
  documented as operator-driven).
- Tests: `test_sync.py` (inject + skip-when-undeclared), `test_forms.py` (widget +
  persist + default-empty), `test_models.py` (allowlist accept + non-list reject),
  `test_query_registry.py` (feature-tags seeds `sync_device_tags`; the existing
  declared-param contract test now also guards it), `test_dependency_preview_summary.py`,
  `test_jobs.py`.

## Approach
Reuse the existing device-tag scope machinery rather than adding a per-sync query
switch: the selection is a Source parameter (like `device_tag_include_tags`),
carried on `ForwardQueryContext`, and injected into the query through the same
`_apply_context_tag_parameters` path the shard keys use. The query filters
`device.tagNames` by the `sync_device_tags` list, so the same `extras.taggeditem`
binding applies the chosen tags. Empty selection ⇒ empty list ⇒ zero rows.

Normalize the model-result to a dict at the summary boundary (every other consumer
already uses `as_dict()`/attributes — verified: this was the only `.get()` on the
raw object). Guard the overlay enqueues with the existing
`sync.jobs.filter(status__in=[PENDING, RUNNING])` pattern so at most one overlay of
each kind is queued per sync.

## Validation
Full suite on NetBox **4.6.4** (dev container rebuilt). Live NQE run of the rewritten
feature-tags query: a two-tag selection returned exactly the matching device/tag
rows with correct slugs; empty selection returned zero rows. Injection unit tests
(sorted selection injected only when the query declares the parameter). Form
persist + validation round-trip. Hermetic dependency-preview summary repro
(dataclass no longer raises; estimated_changes derived). Pile-up guard test.
Lint/harness/sensitive.

## Rollback
Revert the surfaces; `pip install forward-netbox==2.2.4`. The new Source parameter
is additive and ignored by 2.2.4, so a downgrade leaves stored sources valid.
NetBox 4.6.3 still works functionally but is no longer the tested/declared minimum.

## Decision Log
- Sync Device Tags as a Source parameter (not a per-sync toggle or a separate
  query per tag): it is durable config like the include/exclude scope, reuses the
  param-injection path, and needs no query-registry surgery.
- Replace the `Prot_BGP` hardcode rather than add alongside: only Blake uses this
  map today and BGP tagging belongs in the routing plugin; keeping the hardcode
  would double-emit once he selects tags.
- Fix the summary at the boundary via `as_dict()` rather than changing
  `model_results` to dicts (would ripple through every consumer that uses
  attributes).
- Pile-up guard over removing default-on: Blake wants the parent link populated;
  the guard keeps it default-on without the backlog.
- Require 4.6.4 (not just test it): testing a version we don't declare is
  misleading; 4.6.4 is a patch above the prior minimum, low risk.

## Bundled changes
- Feature: operator-selectable **Sync Device Tags** on the Source — choose which
  Forward device tags (e.g. `Mgmt_*`) become NetBox device tags. Replaces the
  hardcoded `Prot_BGP` feature tag. The org-published query is republished with the
  new signature at release (existing sources with no selection sync no feature tags).
- Fix: "Preview dependencies" no longer errors — a per-model summary treated the
  fetch result as a dict; it now reads the dataclass correctly.
- Fix: the automatic "link vsys/vdom parents" job no longer piles up in pending —
  the post-sync overlays skip enqueuing when one of the same kind is already
  queued or running for the sync.
- Change: tested and required NetBox version is now 4.6.4. Drop-in from `2.2.4`.
