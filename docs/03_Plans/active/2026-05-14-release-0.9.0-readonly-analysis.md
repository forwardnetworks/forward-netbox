# 0.9.0 Read-Only Analysis Release

## Goal

Ship the next major release as a read-only analysis layer around the existing
NetBox-native sync flow. The release should surface Forward-native analysis
results and workload previews alongside sync state while keeping mutation on
the Branching path and leaving predict disabled for now.

## Constraints

- Keep NQE as the normalization and source-of-truth layer for row shape.
- Keep NetBox-native objects and Branching-native execution as the mutation path.
- Do not add a separate query authoring system in NetBox.
- Do not enable predict or add a predict mutation path in this release.
- Preserve the current sync workflow, including query-path binding, query-id
  resolution, and branch-backed review.
- Keep future analysis features read-only unless a separate implementation plan
  explicitly adds mutation behavior.
- Do not commit customer identifiers, network IDs, snapshot IDs, screenshots,
  or credentials in examples, tests, or docs.

## Approach

Build the release in three read-only tranches and keep predict out of scope.

### 1. Analysis surfaces on sync records

Attach Forward-native analysis results to existing sync and ingestion objects so
operators can inspect:

- intent verification results
- path verification and search results
- security posture results for firewall and enforcement-point analysis
- routing and reachability warnings
- model-scoped diagnostics for a sync run

These results should remain attached to the existing sync flow instead of
creating a new navigation model.

### 2. Workload and blast-radius preview

Expose a preview layer that explains:

- what objects would be affected
- how large the resulting workload would be
- whether the run should stay on Branching or use fast bootstrap for the seed
- whether the workload should be split or delayed

Fast bootstrap remains the initial seed path only. A completed fast-bootstrap
run can establish the baseline for later Branching diffs, but only after the
baseline contract is satisfied and only for later snapshots.

### 3. Lifecycle enrichment follow-up

Lifecycle enrichment remains a roadmap item, but the 0.9.0 release does not
yet ship a trustworthy NetBox-native source for support-window or end-of-life
advisory data. Keep it deferred until we have a real source contract.

- Model tests for any new persisted metadata.
- API tests for new analysis and workload-preview endpoints.
- UI tests or Playwright coverage for the operator-visible surfaces.

## Rollback

Remove the new read-only analysis surfaces and leave the core sync path
unchanged. Existing sync records, ingestion history, and Branching behavior
should remain intact.

## Decision Log

- Chosen: keep predict out of the 0.9.0 scope because the current objective is
  to harden and expose read-only analysis around the existing sync model.
- Chosen: make fast bootstrap a baseline seed path, not a parallel diff engine.
- Chosen: ship workload preview now and defer lifecycle enrichment until a
  real inventory-support source contract is available.
- Rejected: introducing a separate query authoring system in NetBox because the
  Forward NQE contract is the source of truth.
- Rejected: turning lifecycle enrichment into a mutation path because it should
  be advisory context only.
