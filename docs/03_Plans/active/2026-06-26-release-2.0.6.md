# Release 2.0.6 ‚Äî fix pernicious FHRP-group sync churn (shared VIP)

**Date:** 2026-06-26

## Goal
Stop the recurring "add + remove N FHRP groups every sync" churn Partner reported
(Ingestion 2658: Deleted=13, `ipam.fhrpgroup` 0.0%). Make FHRP-group sync
idempotent.

## Root cause (probe-confirmed)
A live NQE probe of `forward_hsrp_groups.nqe` against the operator's network
found that **a fixed set of virtual IPs are each shared by two different HSRP
`group_id`s** (the same VIP reported under two group numbers) ‚Äî matching the
operator's recurring Deleted count.

`_ensure_fhrp_vip` looked up the VIP IPAddress globally by `(net_host, vrf)`. The
first group claims the VIP; the second group with the same VIP hit the
"vip-conflict" branch and returned `False`, so `apply_ipam_fhrpgroup` saw
`group_created and no assignment` and **deleted the just-created group**. Every
sync re-created and re-deleted those 13 second-groups.

## Constraints
- No schema/migration change. NetBox attaches a VIP IPAddress to exactly one
  FHRP group, so the shared VIP can only live on one of the two groups.
- Idempotent: re-sync must not delete/recreate; same PKs.

## Touched Surfaces
- `forward_netbox/utilities/sync_ipam.py` ‚Äî `_ensure_fhrp_vip`: when the VIP is held by ANOTHER FHRP group, return `True` (group + assignment still persist; no duplicate VIP) instead of `False`; only the non-FHRP-object case stays a hard skip. `delete_ipam_fhrpgroup`: only delete the VIP if it's assigned to THIS group (don't delete the other group's shared VIP).
- `forward_netbox/utilities/sync.py` ‚Äî add `shared-vip` to `SKIP_WARNING_ROLLUP_REASONS`.
- `forward_netbox/utilities/sync_reporting.py` ‚Äî reason‚Üítemplate map for rollup summaries (module-bay message unchanged; new shared-vip message).
- Tests: `test_sync.py` shared-VIP idempotency + delete-safety.

## Approach
A VIP shared by two distinct FHRP groups is legitimate. Keep both groups and
their interface assignments; leave the VIP IPAddress on the first group. The
second group persists (found by `(protocol, group_id, name)` coalesce on
re-sync), so it is no longer created+deleted each run. Deleting a shared-VIP
group no longer strips the other group's VIP. The one-time informational notice
is rolled up to a single line.

## Validation
Unit test: two groups share a VIP ‚Üí both persist, VIP attached once; re-apply
keeps the SAME PKs (no churn); deleting the second group leaves the first group's
VIP intact. Full suite. Probe confirmed the 13 shared VIPs.

## Rollback
Localized to `_ensure_fhrp_vip` + `delete_ipam_fhrpgroup` + the rollup message.
Reverting restores 2.0.5; no schema/data change.

## Decision Log
- Keep both groups (VIP on the first) rather than creating a duplicate VIP
  IPAddress: NetBox's unique-IP guard would reject the duplicate, and a VIP
  attaches to one group anyway. This preserves the most data while being
  idempotent.
- Guard the VIP delete by assigned-group so a shared VIP is never orphaned when
  one of its groups is removed.
