## API/NQE Work Covered

This plan includes the API/NQE issues raised by the ORG usage review:

- Prefix NQE calls now use `forward_netbox_shard_keys` parameters for `ipam.prefix` instead of repeated per-prefix column-filter calls.
- ORG org prefix queries were updated and validated for empty and scoped parameter lists.
- Query maps can carry default NQE parameters even when bound by `query_id` or `query_path`.
- The `ipam.prefix` Branching fetch contract now uses `fetch_mode=nqe_parameters`.
- Existing model fallback artifacts are reused across prefix shards, avoiding repeated broad fallback calls.
- `EQUALS_ANY` value-required errors now retry as single-value `DEFAULT` partitions instead of recursive split spam.
- Forward API request pacing is available as the source parameter `api_requests_per_minute`.
- Forward SaaS sources default to `1800` requests/minute, below the SaaS hard-block threshold of `2000` requests/minute per user.
- The interface NQE now walks device interfaces once instead of three times for ethernet, loopback, and other interface types.
- Query-fetch thread workers close Django DB connections before and after worker jobs, preventing long plan-only runs from accumulating idle PostgreSQL sessions until the local database rejects new connections.
- Smoke/matrix execution now persists `--max-changes-per-branch` into the NetBox `ForwardSync.parameters` record, so the configured NetBox shard size and the runtime ledger agree.
- The local release-validation smoke source is tuned with:
  - `query_fetch_concurrency=2`
  - `nqe_page_size=5000`
  - `api_requests_per_minute=1800`

## Forward API Rate-Limit Proof

The Forward API limiter is implemented at the central `ForwardClient._request()` path, so it applies to all Forward HTTP calls made through the plugin client, not only NQE calls.

Current proof:

- `ForwardClient._request()` calls `_throttle_request()` immediately before each `httpx.Client.request(...)`.
- Retries are throttled too, because `_throttle_request()` runs inside the retry loop.
- The local release-validation source resolves to `api_requests_per_minute=1800`, which produces a minimum request interval of `0.03333333333333333` seconds.
- The SaaS platform limit being protected is `2000` requests/minute per user. Exceeding it can return HTTP `429 Too Many Requests` and block the user account for 5 minutes.
- This limiter is a SaaS protection guardrail, not a capacity SLA. Custom/on-prem sources default to disabled; on-prem rollout is future work.
- The local NetBox cache backend is Redis: `django_redis.cache.RedisCache` at `redis://redis:6379/1`.
- The limiter stores lock/timestamp state in Django cache, so this local runtime shares limiter state across NetBox worker processes via Redis.
- The limiter key hashes `base_url + username`; raw usernames are not stored in the cache key.
- Source-backed smoke/matrix execution can use an existing local `ForwardSource` without exporting credentials into the host shell.
- The current release-validation source uses `nqe_page_size=5000`, reducing Forward NQE pagination calls versus the earlier `500`-row page setting while keeping memory within local headroom.

## Rollback

- Revert the prefix NQE parameterization and query registry changes to return prefix maps to their previous unparameterized behavior.
- Remove or set `api_requests_per_minute=0` on affected local Forward sources to disable client-side request spacing.
- Restore the previous ORG org query definitions if saved-query parameterization causes a live issue.
- Cancel or reconcile local smoke execution runs created only for this validation tranche.

## Decision Log

- Chose query parameters for `ipam.prefix` instead of repeated `EQUALS_ANY` column-filter calls because ORG's logs showed many repeated NQE calls for the same query with different prefix values.
- Chose source-level `api_requests_per_minute` because the rate-limit risk is per Forward user and can be triggered by regression runs beyond prefix NQE alone.
- Chose a default-disabled RPM limiter so existing deployments keep current behavior unless an operator explicitly opts in.
- Chose to cancel the old local release-validation smoke run instead of resuming it because its next planned step still used the obsolete `nqe_column_filter` fetch mode.
- Chose to require fresh release-validation matrix evidence before 1.1 because the current artifact was generated around the transition and contains failed required steps.
- Chose `nqe_page_size=5000` for the release-validation release source because page size 500 produced excessive paginated Forward NQE calls on large models without improving local memory enough to justify the API overhead.
- Chose single-pass interface NQE because `dcim.interface` was the final large model after prefix and page-size tuning.
