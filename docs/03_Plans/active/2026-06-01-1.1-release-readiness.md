## API/NQE Work Covered

This plan includes the API/NQE issues raised by the customer usage review:

- Prefix NQE calls now use `forward_netbox_shard_keys` parameters for `ipam.prefix` instead of repeated per-prefix column-filter calls.
- field-validation org prefix queries were updated and validated for empty and scoped parameter lists.
- Query maps can carry default NQE parameters even when bound by `query_id` or `query_path`.
- The `ipam.prefix` Branching fetch contract now uses `fetch_mode=nqe_parameters`.
- Existing model fallback artifacts are reused across prefix shards, avoiding repeated broad fallback calls.
- `EQUALS_ANY` value-required errors now retry as single-value `DEFAULT` partitions instead of recursive split spam.
- Forward API request pacing is available as the source parameter `api_requests_per_minute`.
- Forward SaaS sources default to `1800` requests/minute, below the SaaS hard-block threshold of `2000` requests/minute per user.
- The interface NQE now walks device interfaces once instead of three times for ethernet, loopback, and other interface types.
- Query-fetch thread workers close Django DB connections before and after worker jobs, preventing long plan-only runs from accumulating idle PostgreSQL sessions until the local database rejects new connections.
- Smoke/matrix execution now persists `--max-changes-per-branch` into the NetBox `ForwardSync.parameters` record, so the configured NetBox shard size and the runtime ledger agree.
- The local release-validation smoke source is tuned with:
  - `query_fetch_concurrency=2`
  - `nqe_page_size=5000`
  - `api_requests_per_minute=1800`

## Forward API Rate-Limit Proof

The Forward API limiter is implemented at the central `ForwardClient._request()` path, so it applies to all Forward HTTP calls made through the plugin client, not only NQE calls.

Current proof:

- `ForwardClient._request()` calls `_throttle_request()` immediately before each `httpx.Client.request(...)`.
- Retries are throttled too, because `_throttle_request()` runs inside the retry loop.
- The local release-validation source resolves to `api_requests_per_minute=1800`, which produces a minimum request interval of `0.03333333333333333` seconds.
- The SaaS platform limit being protected is `2000` requests/minute per user. Exceeding it can return HTTP `429 Too Many Requests` and block the user account for 5 minutes.
- This limiter is a SaaS protection guardrail, not a capacity SLA. Custom/on-prem sources default to disabled; on-prem rollout is future work.
- The local NetBox cache backend is Redis: `django_redis.cache.RedisCache` at `redis://redis:6379/1`.
- The limiter stores lock/timestamp state in Django cache, so this local runtime shares limiter state across NetBox worker processes via Redis.
- The limiter key hashes `base_url + username`; raw usernames are not stored in the cache key.
- Source-backed smoke/matrix execution can use an existing local `ForwardSource` without exporting credentials into the host shell.
- The current release-validation source uses `nqe_page_size=5000`, reducing Forward NQE pagination calls versus the earlier `500`-row page setting while keeping memory within local headroom.

## Release Notes Draft

Forward NetBox `1.1.0` focuses on reducing Forward SaaS API/NQE pressure while preserving NetBox-native Branching behavior.

1.1.0 changes relative to 1.0.0:

- Adds source-level Forward API pacing. SaaS sources default to `1800` requests/minute, below the Forward SaaS `2000` requests/minute per-user hard-block threshold.
- Applies pacing at `ForwardClient._request()`, so retries and non-NQE Forward API calls are paced too.
- Adds sanitized Forward API usage accounting in job data:
  - HTTP attempts
  - successes and failures
  - status classes
  - `429` count
  - retry count
  - throttle sleep seconds
  - NQE query/diff calls
  - NQE page counts
- Changes `ipam.prefix` shard fetches from repeated Forward column filters to `forward_netbox_shard_keys` NQE parameters.
- Seeds built-in IPv4 and IPv6 prefix maps with default empty shard parameters.
- Supports NQE parameters on maps bound by query ID or repository path.
- Uses a single-pass interface NQE instead of separate ethernet/loopback/other interface walks.
- Persists configured `max_changes_per_branch` into sync parameters when smoke/matrix runs create or update a local sync.
- Adds tests proving Forward NQE diffs update existing rows, including rows whose identity key changes.
- Adds a focused prefix-only fresh-sync regression for the field-reported 1.0.0 no-prefix/no-op symptom.
- Adds an isolated test-runtime PostgreSQL readiness wait so one-off isolated NetBox test containers do not race database startup.

Compatibility notes:

- 1.0.0 prefix maps used the older unparameterized prefix query path. A 1.0.0 prefix-only sync that completes with no prefix work does not prove the 1.1.0 prefix path is broken.
- 1.1.0 prefix maps and saved-query bindings should point at the updated parameterized query definitions. Existing saved query IDs must be updated or rebound before relying on the 1.1.0 prefix optimization.
- The local release validation has passing evidence for both NetBox 4.5.9 and 4.6.1 behavior, but the final CI-equivalent pass in this workspace used `NETBOX_VER=v4.6.1` because the shared local database is already on the 4.6 schema shape.

### 1. Stabilize Isolated Test Runtime Startup

### 3. Add Diff Utilization Gates

Priority: high.

Problem:

- The best steady-state API reduction comes from using Forward NQE diffs instead of full model queries.
- The code already supports diff execution for eligible query-id/query-path maps, but release validation should fail or warn if too many eligible models fall back to full queries.
- A low diff-utilization run can look healthy functionally while putting unnecessary load on Forward SaaS.

Proposed implementation:

- Expand existing scale benchmark diff metrics into a release gate.
- For each execution step, record:

### 4. Expand Parameterized Shard Fetches Beyond Prefixes

Priority: medium-high.

Problem:

- The urgent 1.1 fix was `ipam.prefix`, where repeated column-filter calls were producing excessive NQE activity.
- Other high-volume model families may still make multiple scoped fetches when a single saved query with parameters could do the same work with fewer Forward calls.
- The main opportunity is not just fewer calls; it is more predictable query shape and better saved-query behavior in the Forward org repository.

Candidate model families:

- `dcim.interface`
- `dcim.cable`
- `ipam.ipaddress`
- `dcim.inventoryitem`
- `dcim.module`
- `extras.taggeditem`

Proposed implementation:

- Audit `SHARD_FETCH_MODEL_CONTRACTS` and live matrix evidence to identify models with high shard count and repeated `nqe_column_filter` fetches.
- For each candidate, decide whether parameters are safe and natural:
  - device-name scoped parameter list
  - canonical cable endpoint device list
  - prefix/address scope list
  - inventory parent-device scope list
- Update shipped NQE with default-empty parameter handling, matching the prefix pattern.
- Update query registry metadata so built-in maps seed default parameters.
- Update branch fetch contracts to use `fetch_mode=nqe_parameters` where appropriate.
- Publish or document org-query update requirements for saved query users.

Validation:

- Unit tests for each new parameterized map:
  - empty parameter list returns broad/full behavior when expected
  - scoped parameter list returns only scoped rows
  - query registry rejects unregistered shard parameters
  - fetch contract passes parameters instead of column filters
- Field-scale evidence should compare per-model call counts before and after.

Acceptance criteria:

- Candidate models show fewer Forward NQE calls or fewer fallback calls in release evidence.
- Saved query IDs remain compatible with the new parameter model.

### 6. Add Adaptive Runtime Tuning Guardrails

Priority: medium.

Problem:

- The release-validation source is currently tuned manually with `query_fetch_concurrency=2`, `nqe_page_size=5000`, and `api_requests_per_minute=1800`.
- Fixed values are predictable, but they do not react to local database pressure, Forward retry behavior, high latency, or low memory headroom.
- Manual tuning is easy to forget when a user moves between local Docker, larger on-prem runtimes, or SaaS validation runs.

Proposed implementation:

- Keep explicit configured values as hard bounds.
- Add optional adaptive behavior that can reduce pressure during a run:
  - lower query-fetch concurrency if PostgreSQL connection pressure is high
  - lower page size if memory or row materialization warnings appear
  - increase throttle sleep or pause retries after repeated transient Forward errors
  - avoid increasing above configured safe maximums during release validation
- Start with observability-only mode:
  - report "would reduce concurrency from X to Y"
  - report "page size appears high for model Z"
  - report "Forward transient error rate suggests backing off"
- Enable active adjustment only after evidence shows the recommendations are stable.

Likely code surfaces:

- `forward_netbox/utilities/query_fetch_execution.py`
- `forward_netbox/utilities/runtime_capacity.py` or current capacity-review command surfaces
- `tasks.py` release/runtime evidence commands
- `docs/00_Project_Knowledge/runtime-tuning-runbook.md`

Validation:

- Unit tests for recommendation logic using synthetic metrics.
- Scenario test with simulated high DB connection count.
- Scenario test with simulated repeated Forward transient failures.
- Evidence artifact showing recommendations without changing runtime behavior in observation-only mode.

Acceptance criteria:

- Release evidence reports whether runtime settings were within safe headroom.
- The plugin can distinguish "slow because doing useful fetch work" from "slow because local runtime pressure is limiting progress."
- Active adjustment remains opt-in until proven.

## Decision Log

- Chose query parameters for `ipam.prefix` instead of repeated `EQUALS_ANY` column-filter calls because customer logs showed many repeated NQE calls for the same query with different prefix values.
- Chose source-level `api_requests_per_minute` because the rate-limit risk is per Forward user and can be triggered by regression runs beyond prefix NQE alone.
- Chose a default-disabled RPM limiter so existing deployments keep current behavior unless an operator explicitly opts in.
- Chose to cancel the old local release-validation smoke run instead of resuming it because its next planned step still used the obsolete `nqe_column_filter` fetch mode.
- Chose to require fresh release-validation matrix evidence before 1.1 because the current artifact was generated around the transition and contains failed required steps.
- Chose `nqe_page_size=5000` for the release-validation release source because page size 500 produced excessive paginated Forward NQE calls on large models without improving local memory enough to justify the API overhead.
- Chose single-pass interface NQE because `dcim.interface` was the final large model after prefix and page-size tuning.
