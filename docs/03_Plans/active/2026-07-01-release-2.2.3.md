# Release 2.2.3 — field-feedback fixes + two small features

**Date:** 2026-07-01

## Goal
Bundle the field-feedback items from the customer into one patch:
1. Delete-count labeling — the per-model panel showed a PLANNED delete count
   labeled "deletes" (confusing "deleted N but change log empty").
2. vsys/vdom Parent Device populated by default (was opt-in, blank on every
   virtual context).
3. `forward_apply_identity_audit` pinpoints the "1 created + 1 deleted" churn
   object (pair create↔delete + name the key diff; cover dcim.device).
4. Skip re-importing VRFs that have no prefixes/IPs (139 empty VRFs churning).
5. Per-device CVE details — list the actual CVE IDs behind the exposure count.
6. Clarify the "unverified" query-ID health status (cosmetic; reads as an error).

## Constraints
- Patch scope: no destructive re-model. (VDC modeling of vsys/vdom is deferred to
  its own release — it converts existing device rows + reshapes interfaces.)
- Two additive migrations only (`0029` vsys CF from 2.2.x branch, `0030` cve_ids).
- Two `.nqe` changes (VRF filter, device-analysis CVE ids) → the bundled queries
  must be re-published to the Forward org to take effect on live syncs.

## Touched Surfaces
- `templates/.../inc/execution_insights.html` — relabel `delete_count` "planned".
- `jobs.py` + `utilities/model_validation.py` — vsys parent-link default-on
  (opt-out `auto_link_vsys_parents=False`); allowlist the three overlay keys.
- `utilities/apply_identity_audit.py` — `churn_pairs` + `dcim.device` coverage.
- `queries/forward_vrfs.nqe` — only emit a VRF with an addressed subinterface.
- `queries/forward_device_analysis.nqe` + `models.py` (cve_ids JSONField) +
  migration `0030` + `utilities/device_analysis.py` + `inc/device_analysis_panel.html`
  — per-device CVE ID list (linked to NVD).
- `utilities/query_binding_resolution.py` + `forwardsync_health.html` — friendly
  `status_label` for the info-level `direct_query_id_unverified` badge.
- Tests across `test_log_export`, `test_apply_identity_audit`, `test_jobs`,
  `test_models`, `test_scope_module_ui`, `test_execution_insights_delete_label`.

## Approach
All additive/relabel/filter changes; no data destruction. The VRF filter mirrors
how the prefix/IP queries associate addressing to a VRF, so only genuinely-empty
VRFs are skipped (validated live: 418 → 244, 174 empty dropped). CVE ids reuse
the existing device-analysis pipeline (query already counted them; now emits the
list, bounded to 200).

## Validation
Full suite on NetBox 4.6.3. NQE linter on both changed queries (0 errors). Live:
empty-VRF drop count; device.cveFindings shape. Lint/harness/sensitive. After
release, re-publish the bundled queries to the Forward org.

## Rollback
Revert the surfaces; `pip install forward-netbox==2.2.2`. Migrations 0029/0030
are additive (reversible).

## Decision Log
- Keep the vsys CF-link (default-on) and DEFER VDC: VDC is a destructive re-model
  (converts ~391 devices, reshapes interfaces) — not patch-appropriate.
- Skip empty VRFs at the query (don't import) rather than post-sync prune: a prune
  would delete them but Forward re-imports every sync; the query filter stops the
  re-import at the source.
- Bound stored cve_ids at 200; cve_count stays authoritative for the total.

## Bundled changes
- Fix: the per-model execution panel labels its delete number "delete(s) planned"
  (Forward-diff intent computed before apply); the applied deletes are the change
  log / num_deleted. Items already absent plan a delete but apply none, so these
  legitimately differ.
- Feature: vsys/vdom firewalls get their physical chassis linked in the
  `Parent Device` custom field automatically after every sync (opt out with
  `auto_link_vsys_parents=False`).
- Feature: Forward no longer imports VRFs that have no prefixes or IP addresses,
  so empty VRFs stop re-appearing every sync. (Re-publish the bundled queries.)
- Feature: the device Forward Analysis panel now lists the actual CVE IDs behind
  the CVE-exposure count, each linked to the NVD detail page.
- Diagnostic: `forward_apply_identity_audit` pairs each stale NetBox object with
  the Forward row most likely to be the same object and names the exact key
  disagreement, and covers `dcim.device`.
- Polish: the query-ID health status shows "Org-managed (pinned)" instead of the
  raw "unverified" (it is expected and non-blocking for org-managed query IDs).
  Drop-in from `2.2.2`.
