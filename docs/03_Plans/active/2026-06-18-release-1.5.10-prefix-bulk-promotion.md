# Release 1.5.10 — ipam.prefix Bulk-ORM Promotion

## Goal

Promote `ipam.prefix` from the experimental/adapter-required set into the default
bulk-ORM safe set — the last model still off the uniform fast path. This closes
out bulk-ORM coverage for the built-in models so steady-state syncs run the bulk
engine end to end.

## Constraints

- No churn and no behavior change vs the adapter (parity-tested, including the
  NetBox prefix hierarchy `_depth`/`_children`).
- Null-VRF (global table) prefixes must match on re-apply, not duplicate.
- No NQE change → no customer org push.

## Touched Surfaces

- `forward_netbox/utilities/apply_engine_decision.py` — move `ipam.prefix` into
  `BULK_ORM_ENABLED_MODELS`; empty `EXPERIMENTAL_BULK_ORM_MODELS`; drop it from
  `ADAPTER_REQUIRED_MODELS` and `ADAPTER_MODEL_BLOCKERS`.
- `forward_netbox/utilities/apply_engine_bulk.py` — route `ipam.prefix` to the
  per-object tree apply; `nullable_lookup_fields` config; lookup-key builders gain
  a null sentinel + canonical-CIDR normalization (`_normalize_lookup_component`).
- `docs/02_Reference/apply-engine-model-matrix.md` — prefix moves to the safe set.
- Tests: `test_bulk_adapter_parity` (`test_prefix_bulk_matches_adapter`, captures
  `_depth`); `test_sync` (default-bulk, null-VRF no-duplicate, bulk apply);
  `test_health` comment.
- Version bump + 3 release tables.

## Approach

Three blockers solved:

1. **Hierarchy `_depth` parity.** NetBox maintains `_depth`/`_children` via the
   `handle_prefix_saved` post_save signal (not in `Prefix.save()`), and the ingest
   signal suppression does not touch it. Routing prefix through
   `bulk_orm_apply_tree_models` (which saves per object, not `bulk_create`) lets
   the signal fire exactly as in the adapter. Parity test captures `_depth`.
2. **Null-VRF identity.** The composite `(prefix, vrf)` lookup key returned `None`
   when vrf was null, so an existing global prefix never matched → duplicate
   create. `nullable_lookup_fields=("vrf",)` makes the builders encode null vrf as
   a sentinel so the key is stable and matchable.
3. **Typed-field normalization.** NetBox stores `prefix` as canonical CIDR (host
   bits cleared on save); incoming rows are raw strings. The lookup key now
   canonicalizes `prefix` on both sides via netaddr so it matches regardless of
   input form.

## Validation

- `forward_netbox.tests.test_bulk_adapter_parity` (prefix parity incl. `_depth`).
- `forward_netbox.tests.test_sync` prefix decision/apply/null-VRF tests.
- `forward_netbox.tests.test_apply_engine` (VRF-no-clobber via the new tree path).
- `scripts/tests/test_apply_engine_model_matrix` drift guard.
- Full suite; local CI mirror; GitHub CI both matrices.

## Rollback

Restore `ipam.prefix` to `EXPERIMENTAL_BULK_ORM_MODELS` + `ADAPTER_REQUIRED_MODELS`
(+ blocker), revert the routing/lookup-key changes and the matrix doc, drop the
new tests. No schema/data impact.

## Decision Log

- Tree path (per-object save) over `bulk_create`: required so the prefix hierarchy
  signal runs — `bulk_create` would leave `_depth` stale. Accepts per-row saves
  for prefix (a low-volume model relative to interfaces/IPs).
- Generic nullable-field + typed-normalization hooks in the lookup-key builders
  rather than a prefix-only branch: keeps the builders reusable for any future
  model with a nullable identity component or typed key field.
