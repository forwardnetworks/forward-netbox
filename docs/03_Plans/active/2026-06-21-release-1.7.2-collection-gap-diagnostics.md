# Release 1.7.2 — Collection-Gap Diagnostics (2026-06-21)

## Goal

Make Forward collection gaps self-diagnosable inside NetBox and stop a failed
APIC from silently pruning ACI inventory. Operators should see *why* in-scope
devices are backfilled (and how stale / how fast-growing) without running a
manual Forward API probe.

## Constraints

- No customer identifiers, network IDs, snapshot IDs, or raw rows in the diff,
  tests, or docs.
- Backward compatible: reconciliation payloads without the new fields still work.
- ACI delete behavior change must not break existing syncs (ACI maps are new in
  1.7.0, so there is no established auto-delete behavior to preserve).

## Bundled changes

1. **Per-reason backfill breakdown + staleness** — Scope Reconciliation now
   groups backfilled devices by Forward collection error (AUTHENTICATION_FAILED,
   CONNECTION_TIMEOUT, INCOMPLETE_SETUP, STATE_COLLECTION_FAILED) and shows each
   device's stale age, so the "why" is visible without a manual probe.
2. **Opt-in auto-tag refresh** — `auto_tag_backfilled` refreshes the
   `forward-backfilled` tag on every sync so the count stops drifting between
   manual refreshes.
3. **Collection-gap growth/trend** — the health signal escalates to a danger
   state with an "Up N / Down N since the previous reconciliation" note when the
   backfilled count moves, derived from the tag job's own history (no new table).
4. **Per-device collection result** — the device-analysis panel shows the
   specific Forward collection error token next to an unreachable device
   (`ForwardDeviceAnalysis.collection_result`, migration 0026).
5. **ACI delete safety valve** — ACI inventory models are no longer auto-pruned;
   a failed APIC empties its fabric query and would otherwise delete every
   bridge domain / L3Out / pod / node. ACI deletes are held back (with a warning)
   unless `aci_allow_deletes` is set.

  per-reason breakdown and parsed staleness.

## Rollback

Revert the release commit. New report keys and the `collection_result` column are
additive; the panels fall back gracefully when absent; the ACI delete valve is a
no-op for non-ACI models and only holds back ACI deletes (no data is lost).

## Decision Log

- **ACI deletes opt-in, not fabric-aware** — ACI delete rows carry only
  tenant/name, so a blanket opt-in is the simplest correct guard.
- **Trend from job history** — the tag job already records `total_backfilled`;
  no dedicated time-series storage is needed.
