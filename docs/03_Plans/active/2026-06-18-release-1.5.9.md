# Release 1.5.9

## Goal

Give operators an easy, native way to see which in-scope devices are
**backfilled** (tagged in scope but not freshly collected in the latest Forward
snapshot). On the CustomerOrg fabric this is the 22-device gap between NetBox (2218) and
the freshly-collected count Rich expects — confirmed live: `latestProcessed` and
`latestCollected` both resolve to snapshot 1319970, and 22 tagged `Prod_Core`
devices are backfilled in it, so no snapshot selector hides them. Operators
asked for a tag they can filter on.

## Constraints

- Native NetBox UX: a tag operators can filter in the standard device list.
- The tag set must stay accurate — a device that collects fresh again loses the
  tag automatically on the next run (no stale tags).
- Mirror the 1.5.7/1.5.8 background-job pattern for the mutating action.
- No NQE change → no CustomerOrg org push.

## Touched Surfaces

- `forward_netbox/utilities/scope_reconciliation.py` — `forward-backfilled` tag
  constants; `tag_backfilled_devices(sync, *, report=None)`; report exposes the
  internal `_present_backfilled` set.
- `forward_netbox/jobs.py` — `tag_forward_backfilled_devices` job runner.
- `forward_netbox/views.py` — `ForwardSyncTagBackfilledView` (POST enqueues the
  job); the Scope Reconciliation GET passes the tag-filtered device-list URL.
- `forward_netbox/templates/forward_netbox/forwardsync_scope_reconciliation.html`
  — Tag backfilled devices button, a tagged-but-backfilled sample card, and a
  link to the filtered device list.
- Tests: `test_scope_module_ui` (tag add/remove + job enqueue).
- Version bump + 3 release tables.

## Approach

Add a maintained NetBox tag `forward-backfilled`. `tag_backfilled_devices`
computes the scope reconciliation, then reconciles the tag's device set to the
current `present_backfilled` set — adds the tag to newly backfilled devices and
removes it from devices that are no longer backfilled — so it is idempotent and
self-healing. A Tag backfilled devices button on the Scope Reconciliation page
enqueues `tag_forward_backfilled_devices` (background job, mirrors prune), and
the page links to `/dcim/devices/?tag=forward-backfilled` so operators see the
list in the normal device table. The sample card lists the names directly.

## Validation

- `forward_netbox.tests.test_scope_module_ui` — tag added to backfilled device,
  removed once it collects fresh; the view enqueues a job whose runner tags.
- Full suite; local CI mirror (pre-commit clean + twice; harness; harness tests;
  py_compile; mkdocs --strict; build). GitHub CI both matrices.

## Rollback

Revert the tag util/job/view/template additions and the version bump. The
`forward-backfilled` tag, if created, is a normal NetBox tag and can be deleted
from the UI. No schema/data migration involved.

## Decision Log

- Tag over a bespoke list view: a tag is filterable, clickable, and composes with
  every existing NetBox device filter — the lowest-friction "where do I see them"
  answer operators asked for.
- Reconcile (add+remove) rather than add-only: keeps the tag truthful as devices
  recover, so it never accumulates stale members.
- Background job for the write: consistent with prune/create-bays and safe if the
  backfilled set is large on a big fabric.
- Selector note: `latestCollected` does not help here because it resolves to the
  same snapshot as `latestProcessed`; the tag surfaces the devices instead.
