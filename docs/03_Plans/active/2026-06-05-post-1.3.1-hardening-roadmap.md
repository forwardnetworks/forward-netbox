# Post-1.3.1 Hardening Roadmap

## Goal

Make the next tranche after `1.3.1` release-focused rather than feature-sprawl
focused. The operator-visible outcome is a sync integration that can prove
bounded Forward API/NQE usage, repeat-sync idempotence, shipped query contract
coverage, and upgrade safety before each release.

## Constraints

- Do not hold the already-published `1.3.1` release for this work.
- Keep NetBox-native and Branching-native behavior as the correctness baseline.
- Keep NQE as the row-shaping source of truth; Python must not fork query
  semantics.
- Do not reintroduce sync-time NQE column-filter shard fetches.
- Do not persist or commit customer identifiers, network IDs, snapshot IDs,
  credentials, screenshots, or raw live rows.
- Prefer release-gate evidence over one-off manual proof.

## Touched Surfaces

- `forward_netbox/utilities/query_registry.py` for a reusable shipped-query
  contract report.
- `forward_netbox/management/commands/forward_architecture_audit.py` for adding
  query-contract coverage to the existing release architecture gate.
- `forward_netbox/tests/test_query_registry.py` and
  `forward_netbox/tests/test_architecture_audit_command.py` for regression
  coverage.
- Future follow-up surfaces for additional tranches:
  `forward_netbox/utilities/sync_reporting.py`,
  `forward_netbox/utilities/execution_telemetry.py`,
  `forward_netbox/utilities/health_summary_blocks.py`, support-bundle
  serialization, and model-specific adapter tests.

## Approach

1. Add a shipped-query contract summary that joins architecture fetch contracts
   to built-in NQE map definitions.
2. Treat every model with `fetch_mode=nqe_parameters` as requiring shipped NQE
   maps that:
   - declare `forward_netbox_shard_keys`
   - seed a default empty `forward_netbox_shard_keys` parameter
   - include an empty-list guard so the query remains runnable from the UI with
     no parameters selected
   - include at least one positive shard-membership predicate so parameterized
     execution actually constrains the NQE work
3. Include that summary in `forward_architecture_audit` and make
   `--fail-on-gap` fail on query-contract gaps.
4. Keep expanding repeat-sync idempotence coverage model by model after the
   query-contract gate lands. Priority order:
   - `dcim.device`
   - `dcim.interface`
   - `ipam.ipaddress`
   - `ipam.fhrpgroup`
   - routing plugin models
   - delete-only and mixed apply/delete shards
5. Add a release-scale fixture path only after the contract/idempotence gates
   are stable. The fixture must be redacted and synthetic enough to live in the
   repo.

## Future Improvements

### API Usage Telemetry Loop

Expose a first-class API/NQE budget report in the UI and support bundle:

- Forward HTTP attempts
- retry count
- 429 count
- NQE query calls
- NQE diff calls
- NQE pages
- throttle sleep seconds
- per-model and per-step fetch mode
- per-step query parameters, excluding raw data

This turns "we think we reduced API pressure" into an operator-visible release
signal.

### Idempotence Regression Matrix

Make repeat-sync no-op behavior a standing release gate. The minimum matrix is:

- fresh sync creates expected rows
- immediate repeat sync has zero unexpected updates
- dependency-only rows do not mutate authoritative models
- delete statistics are counted
- mixed shards keep create/update/delete/unchanged counts separate

### Shipped Query Contract Validator

Every built-in query map should be contract-checked against the architecture
fetch contract. This protects against future regressions where a model is marked
parameterized but the shipped query cannot actually run parameterized or cannot
run unfiltered in the Forward UI.

### Upgrade And Config Compatibility

Add upgrade tests for existing `1.0`, `1.2`, and `1.3` sync configurations:

- stale execution records
- old query maps
- direct query IDs
- repository-path query maps
- old saved parameter defaults

The expected result is greenfield `1.3.1+` behavior after upgrade without
silently reviving legacy column-filter fetches.

### Operator Remediation

When sync configuration is stale, emit exact remediation:

- query map name
- model
- execution mode
- missing parameter
- expected query path or query ID state
- whether the issue blocks bounded Forward API/NQE behavior

### NetBox-Side Performance

Before faster apply engines, reduce redundant NetBox ORM work:

- dependency lookup batching
- per-sync object cache priming
- fewer per-row ORM checks where adapter parity is already proven
- clearer per-model write/read telemetry

## Rollback

Rollback is a normal revert of the hardening commit. Since the first tranche is
read-only reporting plus tests, no data migration or runtime cleanup is needed.

## Decision Log

- Start with query-contract reporting because it is low-risk, catches a real
  class of regressions, and strengthens release gates without changing sync
  mutation behavior.
- Do not start with a large replay fixture; redaction and fixture-shape work is
  valuable but should not block the immediate release-gate improvement.
- Do not add TurboBulk or replica-cache work in this tranche.
