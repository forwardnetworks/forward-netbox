# Forward NetBox 1.4.0 Release Cut

## Goal

Ship `forward-netbox` `1.4.0` as the current release line after the 1.4
feature-complete tranche, and publish the same validated tree to GitHub and
PyPI.

## Scope

- Bump package and plugin version metadata to `1.4.0`.
- Update the release compatibility and install-reference tables.
- Run the full local release gate and package build checks.
- Tag `v1.4.0`, publish the GitHub release, and upload the wheel/sdist to
  PyPI.

## Constraints

- Do not change functional behavior in the release cut.
- Keep customer-identifying content out of tracked files and release text.
- Keep the release notes honest about what was validated locally versus what
  remains future stretch work.

## Touched Surfaces

- `pyproject.toml`
- `forward_netbox/__init__.py`
- `README.md`
- `docs/README.md`
- `docs/01_User_Guide/README.md`

## Approach

1. Bump the package and plugin version metadata to `1.4.0`.
2. Refresh the release compatibility tables and install examples.
3. Run the full repository release gate and build checks.
4. Publish the release artifacts from the validated commit.

## Validation

- `rtk .venv/bin/invoke harness-check`
- `rtk .venv/bin/invoke harness-test`
- `rtk .venv/bin/invoke lint`
- `rtk .venv/bin/invoke check`
- `rtk .venv/bin/invoke scenario-test`
- `rtk .venv/bin/invoke scale-chaos-test`
- `rtk .venv/bin/invoke test`
- `rtk .venv/bin/invoke playwright-test`
- `rtk .venv/bin/invoke docs`
- `rtk .venv/bin/invoke release-dataset-gate --dataset-label=release-smoke`
- `rtk python -m build`

## Rollback

- Revert the version and docs bump if the release gate fails.
- Keep the previous release tagged current if publication fails.

## Decision Log

- Chosen: treat `1.4.0` as a packaging and publication release over the
  already-complete feature tranche.
- Rejected: adding more feature work before release, because the current tranche
  is already feature-complete and the remaining items are stretch work.
