# Release Plan: 0.8.0

## Goal

Prepare `0.8.0` for release with the opt-in fast bootstrap execution backend and LAG cable endpoint hardening while preserving the default NetBox-native Branching workflow.

## Constraints

- Keep NQE as the source of truth for row shape, identity, and normalization.
- Keep Branching as the default and reviewable sync backend.
- Use fast bootstrap only as an explicit operator-selected path for trusted large baselines.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or private screenshots.
- Require local validation and GitHub Actions success before publishing artifacts.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Sync form/model normalization and execution orchestration
- Fast bootstrap execution backend
- Cable sync conflict handling
- Tests, release docs, and operator guidance

## Approach

1. Validate fast bootstrap on a clean local NetBox stack with a bounded live smoke dataset.
2. Keep row validation, query fetching, adapter application, and ingestion issue reporting shared between Branching and fast bootstrap.
3. Record row-level fast bootstrap issues without silently marking the baseline ready.
4. Skip NetBox-invalid LAG cable endpoints as aggregated warnings instead of aborting cable ingestion.
5. Update release metadata and docs for `0.8.0`.
6. Run the full repo gate, commit, push, and verify GitHub Actions before publishing release artifacts.

## Validation

- Clean Docker reset and bounded fast-bootstrap live smoke
- Targeted unit tests for fast bootstrap success/failure and LAG cable skip behavior
- `invoke sensitive-check`
- `invoke harness-check`
- `invoke docs`
- `invoke ci`
- GitHub Actions success on `main`

## Rollback

If local or GitHub validation fails, keep `0.7.1` as the latest published release and revert the `0.8.0` metadata/docs commit. If a release is published and a blocker is found, publish a newer patch release rather than replacing immutable artifacts.

## Decision Log

- Chosen: opt-in fast bootstrap backend because trusted initial baselines need a faster path that avoids branch schema/event overhead.
- Rejected: making fast bootstrap the default because Branching remains the reviewable NetBox-native steady-state path.
- Rejected: a separate importer command because it would bypass the normal Sync UI/API, validation, and ingestion reporting workflow.
