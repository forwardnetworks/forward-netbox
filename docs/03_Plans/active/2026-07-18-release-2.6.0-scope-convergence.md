# Release 2.6.0 - Enterprise Convergence And Ownership

## Goal

Ship 2.6.0 as a production handoff release on NetBox 4.6.5 with
`netbox-branching` 1.1.1. The release must resolve the reported scope, SNMP
endpoint, virtual-parent, module-bay, software-version, CVE/DLM, partial-merge,
and false-drift failure modes with persisted evidence. No known 2.6 correctness
work is assigned to a later release.

## Release Contract

- Exactly one native NetBox Branching branch is created per sync.
- Validation completes before branch provisioning.
- A partial merge records its failed rows, leaves the branch retryable, and
  cannot mark the ingestion merged, baseline-ready, or ownership-complete.
- Missing module bays are created in the active branch before their modules and
  merge natively on the exact runtime matrix.
- Scope, status-tag, and virtual-parent ownership is durable and per sync.
- Ownership claims use the ingestion primary key as generation plus the
  resolved snapshot ID.
- Managed assignments materialize from the union of current claims. The last
  claim controls removal; one sync cannot remove another sync's assignment.
- Parent conflicts remain durable evidence, preserve the current relationship,
  and block ownership finalization.
- Stale overlay jobs mutate nothing and request reconciliation for the latest
  completed generation.
- Sync/source deletion releases claims and rematerializes surviving ownership.
- Existing operator-created virtual contexts are never adopted as plugin-owned.
- Automatic orphan device deletion is unavailable; reviewed manual cleanup is
  the only deletion path.
- Drift, health, audit, UI, and support exports cannot report convergence while
  merge or required ownership reconciliation is incomplete.
- The package contains no selectable direct-write backend and creates no
  execution-ledger rows.

## Constraints

- Require NetBox 4.6.5 and exactly `netboxlabs-netbox-branching` 1.1.1 in
  package resolution, CI, development, runtime diagnostics, and release proof.
- Preserve unrelated operator and feature tags.
- Do not infer historical per-sync claims during migration.
- Keep ownership tables in the main schema; branch schemas fake the ownership
  migrations.
- Serialize ownership mutation with the PostgreSQL advisory lock and row locks.
- Keep normalization in NQE and NetBox mutation in tested adapters.
- Do not commit credentials, customer identifiers, tag values, device names,
  network IDs, snapshot IDs, support rows, private communications, or images.
- Do not publish, tag, or release until the human operator separately
  authorizes it after acceptance evidence is complete.

## Touched Surfaces

- Forward source, sync, ingestion, ownership, identity, and preserved-assignment
  models plus their migrations.
- Single-branch planning, execution, merge finalization, recovery, post-sync,
  scope reconciliation, device identity, virtual-parent, module, DLM, and
  optional-plugin adapters.
- Sync, ingestion, drift, health, ownership, and support-export UI and commands.
- NQE query contracts, dependency ordering, plugin configuration, packaging,
  exact-runtime development images, CI, release tasks, and operator docs.
- Unit, migration, scenario, exact-stack, packaging, sensitive-content, UI,
  scale, failure-injection, and customer-equivalent acceptance tests.

## Approach

1. Pin and validate the exact NetBox and Branching runtime.
2. Add main-schema ownership models for managed tags, per-sync tag claims,
   plugin-owned virtual contexts, per-sync parent claims, and reconciliation
   state.
3. Add generation-stamped post-merge overlays with stale-job rejection,
   catch-up, pending/completed/failed state, and global serialization.
4. Materialize tags and parent relationships from current claim unions,
   including last-claim deletion and conflict failure semantics.
5. Release ownership transactionally on model and queryset sync/source delete.
6. Make migration initialize no claims and adopt no operator virtual contexts.
7. Make partial merge strict and retryable against the complete logical branch
   returned by Branching 1.1.1, rejecting any retry whose logical total changes.
8. Create missing module bays in the branch-native device/module path and prove
   device, bay, and module merge on the exact matrix.
9. Surface incomplete ownership and partial merge in drift, health, audit,
   ingestion issues, templates, and sanitized support output.
10. Add a strict ownership audit that can require zero nonterminal branches.
11. Restore the documented watch and warning-audit commands as read-only
    single-branch tools rather than ledger adapters.
12. Remove contradictory current-path documentation for branch shards,
    direct-write bootstrap, execution-ledger state, and deferred ownership.
13. Delete the no-op execution-ledger module, run/step choices, caller branches,
    empty health/support payloads, and tests for models removed by migration
    `0028`.
14. Remove retired backend constants, aliases, fallback imports, cache paths,
    column-filter fetch contracts, and persisted progress state. Retain only the
    current single-branch contract and its durable ingestion, core Job, branch,
    validation, issue, and ownership facts; migrations may only convert or
    delete retired state once.
15. Promote the module, routing, peering, DLM, and ACI integration paths that
    pass the exact all-plugin matrix to supported status, and state unsupported
    protocol/data boundaries as deliberate product scope rather than beta work.
16. Make a zero-debt architecture audit part of release acceptance: no active
    implementation plan other than this release plan, no no-op compatibility
    module, no current beta/deferred/future-work marker, and no known
    correctness or maintainability item.
17. Fail closed at every operational boundary: encrypt form credentials before
    strict client validation, preserve future scheduled producers, recover dead
    standing schedules from durable intent, enforce exact optional-plugin
    availability during execution, and bind release evidence to its real gate.
18. Make repository publication fail closed: scan current text plus every
    post-baseline changed blob, commit message, and annotated tag message;
    reject unreviewed binary content by exact path and SHA-256; require a
    nonempty private-pattern feed for push and release workflows; and bind the
    tracked history baseline to an external repository trust anchor.
19. Protect `main` and version tags with non-bypassable repository rules and
    require exact-commit NetBox CI plus Python and JavaScript CodeQL checks
    before the release controller can promote or tag a candidate.

## Rollback

### Upgrade Procedure

1. Back up NetBox and quiesce Forward sync workers.
2. Resolve or explicitly preserve every nonterminal Forward branch before
   migration; release acceptance requires zero open branches.
3. Install NetBox 4.6.5, Branching 1.1.1, and the 2.6.0 candidate together.
4. Run migrations and restart NetBox services and workers.
5. Run `manage.py check` and the strict ownership audit.
6. Run every relevant sync to establish current generation claims; migration
   deliberately invents no historical claims.
7. Wait for all required ownership domains to complete before accepting drift
   convergence.

### Rollback Procedure

Rollback across the ownership schema boundary is backup restore, not a mixed
package downgrade. Restore NetBox, plugin, Branching, database, and worker state
as one tested unit. Do not run an older plugin against 2.6 ownership state.

## Decision Log

- Use exactly NetBox 4.6.5, Branching 1.1.1, and Python 3.14 so development,
  CI, packaging, and release evidence prove one runtime instead of a range.
- Use one native branch per sync; rejected direct-write and secondary execution
  ledgers because they create alternate state and merge semantics.
- Persist ownership and device identity in the main schema; rejected inferred
  historical claims because they could transfer control of operator data.
- Finalize merge baseline and ownership registration atomically; rejected
  best-effort post-merge state because it can report false convergence.
- A configured include tag is authoritative plugin-managed scope on migration,
  so migration invents neither claims nor preserved assignments for it. Later
  explicit adoption of a different pre-existing tag records its current
  assignments; explicit operator removal clears that preservation.
- Keep release evidence local and sanitized summaries tracked; rejected raw
  tenant artifacts because repository history is permanent.
- Treat stored schedule intent and canonical optional-plugin capability status
  as execution contracts. A missing worker occurrence is repaired without
  mutating sync state, while a missing or mismatched optional package is blocked
  before any query or adapter execution.
- Bind each release-authorization entry to its canonical command and expected
  evidence vocabulary; rejected generic command-success text because it cannot
  prove customer acceptance, ownership, UI, artifact, scale, or review gates.
- Keep the protected-history baseline externally approved and stable across
  ordinary releases; rejected a candidate-controlled trust anchor because it
  could exclude its own earlier commits from scanning.
- Permit binary documentation assets only by reviewed path and exact SHA-256;
  rejected extension-based or path-only exceptions because changed screenshots
  and non-UTF-8 artifacts are otherwise uninspectable by the text scanner.
- Require all repository-rule checks on the exact candidate SHA before main or
  tag promotion; rejected branch-level workflow success as insufficient proof
  when multiple workflows can run for the same branch.
- Require reviewed production and evidence-only PRs, bind the trusted
  base-branch scanner result to each final candidate SHA, and have the tag
  workflow re-prove exact PR approval, commit verification, and CI/CodeQL
  workflow identities for both main commits. This keeps publication fail-closed
  even where GitHub reports that repository administrators can bypass an
  environment approval in the web UI.
- Establish `security-bootstrap-2.6` as the immutable reviewed root for the
  release workflow, trusted scanner, provenance verifier, reproducible builder,
  and hash-locked toolchain. Execute the verifier from that anchor and reject
  later edits to those files; rejected trusting executable code from the tag it
  is responsible for validating.
- Create bootstrap and version tags only from the frozen protected-main tag
  workflow using an environment-scoped deploy key recognized by creation-only
  rulesets. Human tag pushes and repository-wide key secrets are rejected
  because either can select candidate-controlled release code before anchored
  verification.

## Release Acceptance

1. The exact dependency matrix is installed and reported at runtime.
2. All validation-ladder items pass on the final commit.
3. No migration drift, nonterminal branch, pending/failed required ownership
   reconciliation, merge failure, blocker issue, or sensitive-content finding
   remains.
4. A first baseline creates one branch, merges cleanly, and completes required
   ownership domains for its exact generation.
5. A second sync against the same resolved snapshot reports zero applied,
   created, updated, deleted, and failed changes.
6. Shared managed tags survive until the final claim is removed.
7. SNMP endpoints obey include/exclude scope, CIMC endpoints do not become
   standalone parentless devices, and virtual contexts preserve exact parent
   ownership.
8. Module inventory creates native bays without an operator pre-seed step.
9. CVE/DLM and software-version relationships appear only from the approved
   published query contracts and link to the intended device/software objects.
10. Support output contains actionable aggregate evidence without private
    inventory rows or credentials.
11. Current architecture, code, tests, and documentation contain no known
    technical debt, architecture gap, beta path, or deferred release work.
12. No tag or release is created until the operator explicitly authorizes it.
