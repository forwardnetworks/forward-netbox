# Forward NetBox 1.3 Parameterized Shard Fetch

## Goal

Eliminate default Forward NQE column-filter shard fetches in favor of query-side
NQE parameters for the built-in query set. Parameterized execution should reduce
Forward SaaS backend work by allowing the query itself to short-circuit shard
scope, while preserving current NetBox row shapes and branch shard boundaries.

## Scope

- Add optional shard parameters to built-in NQEs that currently rely on
  `nqe_column_filter` fetch contracts.
- Change default branch fetch contracts to `nqe_parameters` for built-in models
  once their query accepts the required parameter.
- Keep local shard safety filtering even when NQE parameters are used.
- Update query registry tests, branch budget tests, fetch execution tests, and
  user/reference docs.

## Approach

Update the shipped built-ins that already shard by an explicit key so they can
accept `forward_netbox_shard_keys` directly, then switch the default fetch
contracts for those models to `nqe_parameters`. Keep local safety filtering in
the planner so the branch plan still enforces the same row boundaries even when
the query itself performs the narrowing. Treat the parameterized path as the
only supported execution path for the built-ins in this tranche, and keep the
execution and diagnostics aligned with that contract.

## Rollback

Revert the query parameter additions, restore the previous `nqe_column_filter`
fetch contracts, and republish the prior org-repository query revisions. The
branching/state model does not change, so rollback is limited to query source
and fetch-planning behavior.

## Decision Log

- Prefer query-side shard parameters over column-filter fetches because they
  lower backend execution cost without widening the returned row set.
- Keep the shipped built-ins parameter-only so the execution path stays simple
  and the new contract is visible in tests and release evidence.
- Preserve local safety filtering in the planner to avoid moving shard boundary
  logic entirely into Forward.

## Constraints

- Do not broaden Forward pulls as the default replacement for column filters.
- Do not change NetBox row shapes emitted by built-in NQEs.
- Do not include customer identifiers, network IDs, credentials, or screenshots.
- Do not increase branch budgets to hide API/NQE cost.

## Implementation Plan

1. Inventory current `nqe_column_filter` fetch contracts and map each to a
   stable NQE parameter family.
2. Add query-side filters for device-scoped built-in NQEs.
3. Add query-side filters for structured identity built-in NQEs such as site,
   manufacturer, role, platform, device type, device, VLAN, VRF, and OSPF area.
4. Update `shard_fetch_contract()` to prefer `nqe_parameters` for those models.
5. Validate with focused query-registry, branch-budget, query-fetch, and sync
   tests before broader release gates.
