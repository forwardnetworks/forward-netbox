# Release 2.0.8 — progress bars reach 100% on a completed sync

**Date:** 2026-06-26

## Goal
A completed sync that applied/merged cleanly was showing several per-model progress
bars stuck below 100% (Blake: "says it finished but the progress screen shows this"
— `dcim.cable` 50%, `dcim.device` 50%, `dcim.moduletype` 50%, `ipam.fhrpgroup`
71.4%). Make a finished run render every model at 100%.

## Constraints
- Cosmetic/presentation only — no change to apply, merge, statistics accounting, or
  data. The underlying run is correct (status `completed`, validation `passed`,
  "301 applied, no failed").
- Keep in-flight (running) progress accurate — only a terminal-completed job is
  forced to full.

## Touched Surfaces
- `forward_netbox/utilities/ingestion_presentation.py` — `get_statistics` renders
  each model bar at `100.0` when the resolved job status is `completed`; otherwise
  unchanged (`min(current, total) / total * 100`).
- Tests: `test_models.py::ForwardIngestionProgressStatisticsTest` (completed → 100%
  for relationship/two-phase models; running → partial preserved).

## Approach
The bar is `current / total * 100`. For relationship and two-phase models the
bulk-merge `total` counts ChangeDiff rows ("Going to merge 12 changes for
dcim.device") while `current` counts applied objects (6), so a finished model
settles at 50%/71% even though every change merged with zero failures/skips. The
denominator's unit varies per writer (cable+termination, device+primary_ip,
module+moduletype, fhrp group+assignment), so it cannot be normalized reliably
per-model. A completed job has, by definition, applied every change it was going to
— so clamp the bar to 100% on completion. Failures/skips remain visible in
Ingestion Issues and the statistics block; they are not hidden by a full bar.

## Validation
Unit: completed job → 100% for cable/device/fhrp/module; running job → 50% preserved.
Full suite. Lint/harness/sensitive. Page-render of the ingestion progress partial.

## Rollback
Single function; reverting restores `2.0.7` behavior. No schema/data/accounting
change.

## Decision Log
- Clamp on completion rather than re-derive the denominator per writer — the
  per-model `total` semantics differ (relationship vs two-phase), so a unit-correct
  denominator would touch every apply path; the completion clamp is localized and
  correct (finished ⇒ 100%).
- Kept running-state math untouched so in-flight bars stay truthful (still clamped
  to ≤100% for oversized intermediate accounting).
