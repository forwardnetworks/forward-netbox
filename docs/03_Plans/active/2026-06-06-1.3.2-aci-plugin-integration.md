# Release 1.3.2 ACI Plugin Integration Plan

## Goal

Add an optional plugin integration framework, then use it to support
`netbox-cisco-aci` as the first new `1.3.2` target. The user-visible
outcome is that Forward continues to import ACI infrastructure into native
NetBox objects and can optionally project proven ACI domain data into
ACI-specific plugin models without increasing Forward SaaS API/NQE pressure.

This should not be a one-off ACI hook. `forward-netbox` already has optional
plugin behavior for routing/peering-style models, and `1.3.2` should turn that
pattern into a small, native, repeatable architecture for future optional
NetBox plugin integrations.

## Constraints

- Keep native NetBox imports as the default behavior. ACI plugin support must be
  opt-in and disabled unless the target plugin ContentTypes are available.
- Do not make `netbox-cisco-aci` a required dependency for `forward-netbox`.
- Do not add lossy native mappings for ACI policy concepts that NetBox core does
  not model.
- Do not reintroduce sync-time NQE column-filter fetches or per-value NQE
  execution. Any new Forward query path must use saved queries, bounded
  pagination, and parameterized execution where sharding is required.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials,
  screenshots, or raw live rows.
- Use generic Forward signals, APIC/ACI semantics, and model contracts. Do not
  hard-code customer-specific hostnames, tags, tenants, or device names.
- Treat Forward NQE as the source of truth only for fields that Forward exposes
  reliably. For full ACI policy semantics, APIC may remain the more canonical
  source until Forward data shape is proven.
- Keep optional-plugin integrations architecturally uniform. New plugin support
  must go through capability detection, model contracts, query contracts,
  adapter registration, and no-plugin fallback tests.

## Native NetBox ACI Scope

The following ACI-adjacent data should remain in the existing native NetBox
sync path:

- ACI leaf, spine, border leaf, and APIC inventory as `dcim.device`.
- Cisco manufacturer, physical model, role, and platform as `dcim.manufacturer`,
  `dcim.devicetype`, `dcim.devicerole`, and `dcim.platform`.
- ACI-mode NX-OS classification as platform `ACI`, with standalone NX-OS left
  as `NXOS`.
- Interfaces, LAGs, interface MAC addresses, inferred cables, modules,
  inventory items, sites, VLANs, VRFs, prefixes, IP addresses, and native FHRP
  groups.
- Optional tags for lightweight classification, such as `aci`, `aci-leaf`,
  `aci-spine`, `aci-border-leaf`, or `apic`, only when Forward exposes reliable
  role or platform evidence.

These are the safe mappings because NetBox core already has a native object
with a close semantic fit.

## ACI Plugin Scope

The optional ACI plugin path should handle ACI-specific policy/domain objects
that NetBox core does not natively model:

- fabrics
- pods
- ACI nodes linked to native `dcim.device`
- tenants
- ACI VRF policy objects, distinct from native `ipam.vrf`
- bridge domains
- bridge-domain subnets
- application profiles
- EPGs
- ESGs
- contracts
- filters
- L3Outs
- static path bindings
- domains, VLAN pools, AAEPs, and interface policy/profile objects only after
  field shape and identity contracts are proven

This path should target `netbox-cisco-aci` first because it is the active
validated plugin surface. The implementation must keep the adapter boundary
small enough that another adapter, such as `pheus/netbox-aci-plugin`, can be
added later without changing the native sync core.

## Forward/NQE ACI Evidence

The `1.3.2` design should treat ACI data as three different source tiers. This
keeps the first release useful without pretending every APIC policy object is
already a stable public NQE model.

### Tier 1: Typed Native Fields

These are already safe to use for native NetBox sync and for optional ACI
classification:

- `network.devices` platform, OS, OS version, vendor, model, serial, site, and
  interface-derived data.
- Existing native query maps for devices, platforms, device types, interfaces,
  LAGs, VLANs, VRFs, prefixes, IP addresses, MAC addresses, FHRP, cables,
  modules, and inventory items.
- Generic ACI classification signals:
  - APIC OS is APIC.
  - ACI-mode NX-OS uses ACI/NXOS-ACI indicators or ACI NX-OS version trains.
  - Standalone NX-OS remains NXOS.

These fields should stay the default path and should not require an ACI plugin.

### Tier 2: Command Inventory Discovery

Forward exposes collected and selected generated device state through
`device.outputs.commands`. The public command record shape is:

- `command.commandType`
- `command.commandText`
- `command.response`

Source review in `~/src/fwd` shows ACI/APIC states are mapped to NQE command
types such as:

- `CISCO_ACI_EPM_ENDPOINTS`
- `CISCO_ACI_FABRIC_NODES`
- `CISCO_ACI_FABRIC_VRFS`
- `CISCO_ACI_NODE_TYPE`
- `CISCO_ACI_SPINE_INTERFACE_MODE`
- `CISCO_ACI_SPINE_IP_ENDPOINTS`
- `CISCO_ACI_SPINE_TUNNELS_NEXTHOP`
- `CISCO_ACI_TUNNELS_ENDPOINTS`
- `CISCO_ACI_ZONING_FILTER`
- `CISCO_ACI_ZONING_RULE`
- `CISCO_APIC_CONTROLLER_DETAIL`
- `CISCO_APIC_EXT_L3OUT_INTERFACES`
- `CISCO_APIC_SWITCH`
- `CISCO_APIC_VLAN_LEAF`
- `CISCO_APIC_VLAN_STATUS_LEAF`
- `CISCO_APIC_VPC_MAP`

The Forward source also shows only selected generated APIC states are included
in public NQE command outputs. Richer APIC collector/parser internals exist, but
they are not automatically a public NQE contract.

Live validation against an approved field dataset found substantial ACI/APIC
coverage:

- ACI/APIC candidate devices are present at field scale.
- APIC command outputs exist for controller and switch detail.
- ACI-mode leaf/spine command outputs exist for fabric nodes, fabric VRFs,
  zoning rules, zoning filters, endpoint tables, tunnel endpoint tables,
  VLAN leaf/status, and spine endpoint/tunnel data.
- A single bounded page over selected ACI command types completed with one API
  call and one NQE page.

This supports adding a discovery-only `aci.command_inventory` map that reports
device identity, command type, response length, and presence flags. It should
not return raw command responses.

### Tier 3: Command-Output Parsers

If the data is not available as typed NQE rows, command-output parsing is still
an option. It must follow a strict contract:

- Narrow by saved query and command type before reading `command.response`.
- Parse into small normalized rows in NQE when NQE string/pattern functions are
  sufficient.
- If NQE cannot parse the response shape safely, have NQE return only the
  minimal bounded command responses for that command family and parse them in a
  plugin adapter.
- Never select every command response across every ACI device.
- Never persist raw command responses in sync artifacts or support bundles.
- Add response-size guards and skipped-row accounting.

The live field check showed why this matters: some ACI command responses are
small, but others are hundreds of kilobytes to multiple megabytes per device.
Broad raw-response selection is therefore a scaling risk and should not be part
of the `1.3.2` implementation.

### Candidate 1.3.2 ACI NQE Maps

Start with maps that are useful, bounded, and operator-explainable:

- `aci.command_inventory`: discovery only; one row per device and ACI command
  type, with response length and availability.
- `aci.node_inventory`: APIC/fabric node inventory from
  `CISCO_APIC_SWITCH` and/or `CISCO_ACI_FABRIC_NODES`; first candidate for
  optional plugin writes if identity is stable.
  The APIC switch/controller path is now implemented as a separate
  `Forward ACI APIC Nodes` map so controller inventory stays bounded and
  explicit.
- `aci.fabric_vrf_inventory`: discovery from `CISCO_ACI_FABRIC_VRFS`; write
  only after tenant/VRF identity is proven.
- `aci.vlan_bd_epg_inventory`: discovery from `CISCO_APIC_VLAN_LEAF`,
  `CISCO_APIC_VLAN_STATUS_LEAF`, and endpoint data; write only after bridge
  domain and EPG identity is proven.
- `aci.zoning_observation`: discovery from `CISCO_ACI_ZONING_RULE` and
  `CISCO_ACI_ZONING_FILTER`; no contract/filter writes until parsing and
  identity rules are unambiguous.

The first write path should be ACI nodes/fabric inventory. Tenant, bridge
domain, EPG, contract, filter, and L3Out writes should remain gated behind
parser proof and repeat-sync idempotence tests.

## Optional Plugin Integration Architecture

`1.3.2` should define a generic integration shape before adding ACI writes. The
goal is to make optional plugins feel native to NetBox operators while keeping
`forward-netbox` installable without those plugins.

### Integration Registry

Add a small registry of optional integrations. Each integration should declare:

- integration key, such as `aci.netbox_cisco_aci`, `routing.netbox_routing`, or
  `peering.netbox_peering_manager`
- target app labels
- required ContentTypes
- optional ContentTypes
- supported Forward model identifiers
- shipped query maps
- adapter class
- enablement default
- release-contract status

The registry should be read-only metadata plus adapter factories. It should not
import optional plugin models at module import time.

### Capability Detection

Capability detection should be centralized and reusable:

- Use Django app registry and ContentTypes to discover installed plugin models.
- Return a structured capability report with `available`, `missing_required`,
  `missing_optional`, and `unsupported_version` fields.
- Include capability status in `forward_architecture_audit`.
- Surface missing capability reasons in sync issues and support bundles.
- Keep no-plugin startup behavior clean and covered by tests.

### Model Contract

Every optional plugin model needs a model contract before writes are enabled:

- natural identity fields
- required dependency lookups
- allowed create/update/delete behavior
- stale-object behavior
- field normalization rules
- ownership marker, tag, or comment policy
- idempotence expectations
- support-bundle redaction rules

This prevents plugin adapters from becoming ad hoc ORM code with hidden
semantics.

### Adapter Contract

Adapters should expose the same narrow lifecycle:

- `capabilities()` reports installed model support.
- `validate_rows()` catches malformed query rows before ORM writes.
- `plan_dependencies()` resolves native and plugin dependencies in batches.
- `apply_rows()` performs create/update/delete with idempotent comparison.
- `summarize()` returns created, updated, deleted, unchanged, skipped, and
  failed counts.

Adapters should depend on existing sync-runner accounting instead of inventing
new statistics paths.

### Query Contract

Optional plugin queries should use the same release contract as native queries:

- shipped NQE map exists
- saved query ID support is documented
- parameter defaults are present
- UI no-parameter execution works when applicable
- positive shard predicates exist for parameterized maps
- no sync-time column-filter fallback
- aggregate row counts and API/NQE usage are visible in support bundles

### Configuration And UX

Optional integrations should be operator-visible but quiet by default:

- disabled by default unless explicitly enabled by the operator
- show unavailable integrations as unavailable, not broken
- explain missing plugin, missing model, and missing query-map prerequisites
- let native sync continue when optional integrations are unavailable
- keep release compatibility docs explicit about optional plugin requirements

## Approach

1. Add the optional-plugin integration registry. **Status: implemented for the
   initial ACI tranche.**
   - Define the common integration metadata shape.
   - Move existing routing/peering optional-plugin knowledge toward this
     registry where practical, without changing behavior in the first pass.
   - Add architecture-audit output for registered optional integrations.

2. Add a plugin-detection layer. **Status: implemented for the audit report;
   runtime capability surfacing now includes availability and missing-model
   breakdowns.**
   - Detect `netbox-cisco-aci` by installed app label and ContentTypes.
   - Report missing plugin models as disabled optional capability, not as a
     sync failure.
   - Add an architecture audit section that shows ACI plugin availability and
     supported model count.
   - Current implementation reuses ContentType-based seeding, optional model
     resolution, and an installed/available capability report with required
     and optional missing-model breakdowns in the architecture audit,
     support bundle, sync health summary, and health UI, including the actual
     missing model names for operator troubleshooting. A no-plugin seed test
     now proves ACI maps stay skipped when ContentTypes are absent.

3. Add an ACI adapter boundary. **Status: implemented for the expanded 1.3.2
   ACI model surface.**
   - Create a small adapter interface that maps normalized Forward ACI rows to
     plugin model upserts/deletes.
   - Keep plugin-specific imports isolated so NetBox starts cleanly without the
     ACI plugin installed.
   - Model identity must be explicit per object type, such as
     `fabric + pod + node_id`, `tenant + bridge_domain`, or
     `tenant + app_profile + epg`.
   - Current write identities:
     - Fabric: `name`.
     - Pod: `aci_fabric + pod_id`.
     - Node: `aci_pod + node_id`.
     - Tenant: `aci_fabric + name`.
     - ACI VRF: `aci_tenant + name`.
     - Bridge domain: `aci_tenant + name`, with required ACI VRF dependency.
     - Application profile: `aci_tenant + name`.
     - EPG: `aci_tenant + app_profile + name`, with bridge-domain dependency.
     - Contract: `aci_tenant + name`.
     - Filter: `aci_tenant + name`.
     - L3Out: `aci_tenant + name`, with ACI VRF dependency.
     - Static port binding: `aci_endpoint_group + dcim.interface + encap_vlan`.
   - Proven write maps are fabric, pod, node, tenant, VRF, and filter.
   - Bridge domain, application profile, EPG, contract, L3Out, and static port
     binding adapters are present, but the shipped maps remain conservative
     zero-row contracts until bounded parser identity and repeat-sync proof are
     available.

4. Start with read-only capability discovery. **Status: implemented at the map
   contract level; command inventory metadata is now surfaced in the audit.**
   - Add a management command or architecture audit mode that lists available
     ACI plugin models, required fields, and candidate Forward fields.
   - Add ACI command-inventory discovery so operators can see which ACI command
     families exist before enabling plugin writes.
   - Keep discovery outputs to command type, device identity, presence, and
     response-size metadata.
   - Do not write ACI plugin objects until the model contract is proven in
     tests.
   - Current implementation adds architecture-audit registry visibility,
     live-validated ACI NQE row shapes, a surfaced ACI command-inventory
     catalog in the audit output, and a seeded discovery-only ACI command
     inventory query map covering APIC switch/controller, ACI fabric nodes,
     fabric VRFs, zoning, endpoint, tunnel, VLAN, external L3Out, and vPC
     presence.

5. Build the first write path only for high-confidence objects. **Status:
   implemented for fabric/pod/node plus tenant/VRF/filter; contract-only
   surfaces are present for the rest.**
   - Candidate first tranche:
     - ACI node records linked to native devices.
     - Fabric/pod records if Forward exposes stable fabric and pod identity.
     - Tenant/VRF/bridge-domain records only if Forward exposes policy objects
       directly and not inferred approximations.
   - Defer bridge-domain, application-profile, EPG, contract, L3Out, and static
     binding writes until the source data and identity rules are unambiguous.
   - Where ACI data is command-only, prefer NQE-side parsing into normalized
     rows when feasible; otherwise parse narrowly bounded command-family
     responses in the adapter.
   - Current NQE parsing sources: `CISCO_ACI_FABRIC_NODES`,
     `CISCO_ACI_FABRIC_VRFS`, and `CISCO_ACI_ZONING_FILTER`.
   - Current emitted plugin maps:
     - `Forward ACI Fabrics`
     - `Forward ACI Pods`
     - `Forward ACI Nodes`
     - `Forward ACI Tenants`
     - `Forward ACI VRFs`
     - `Forward ACI Filters`
   - Current contract-only plugin maps:
     - `Forward ACI Bridge Domains`
     - `Forward ACI Application Profiles`
     - `Forward ACI Endpoint Groups`
     - `Forward ACI Contracts`
     - `Forward ACI L3Outs`
     - `Forward ACI Static Port Bindings`

6. Keep API/NQE usage bounded. **Status: implemented for current 1.3.2 maps.**
   - Use one saved query per ACI model family, not one query per tenant, EPG,
     or bridge domain.
   - If sharding is required, use `forward_netbox_shard_keys` parameters.
   - Keep UI execution behavior sane by using empty-parameter guards.
   - Do not use sync-time column filters.
   - Do not select broad raw command responses.
   - Add response-size and row-count guardrails for command-output parsers.
   - Add query-contract tests before enabling any map by default.
   - Live validation completed against an approved field dataset:
     fabric/pod/node/tenant/VRF/filter returned expected bounded row shapes.
     Bridge-domain, app-profile, EPG, contract, L3Out, and static-binding maps
     returned zero rows quickly as contract-only maps. A prior zoning-rule
     contract parser timed out under the live bound and was intentionally
     demoted back to a zero-row contract map.

7. Add operator controls. **Status: implemented for map enablement and docs.**
   - Add ACI plugin maps disabled by default.
   - Surface exact prerequisites in configuration docs.
   - Show missing plugin, missing ContentType, missing native dependency, and
     skipped row reasons in sync issues/support bundles.
   - Current implementation seeds the expanded ACI maps disabled by default
     when plugin ContentTypes exist and records missing optional plugin models
     through the same error path used by routing/peering.

8. Validate against the installed ACI plugin workflow.
   - Confirm that a normal native sync continues to classify ACI devices
     correctly.
   - Confirm that enabling the optional ACI maps writes only the intended plugin
     rows.
   - Confirm that a repeat sync is idempotent and produces zero unexpected
     updates.
   - Promote only maps that meet the same live parser and repeat-sync evidence
     bar as fabric/pod/node/tenant/VRF/filter.

## Other 1.3.2 Work To Fold In

### ACI Classification Hardening

- Keep the current generic ACI-mode NX-OS platform normalization in the shipped
  platform and device queries.
- Add regression examples for APIC, ACI-mode NX-OS, and standalone NX-OS.
- Add an operator-facing note that rerunning `dcim.device` is what moves
  existing devices from `NXOS` to `ACI`; rerunning only `dcim.platform` creates
  or updates the platform row but does not reassign existing devices.

### Optional ACI Device Tags

- Add a lightweight native tag map only if Forward exposes reliable role
  signals.
- Candidate tags: `aci`, `aci-leaf`, `aci-spine`, `aci-border-leaf`, and
  `apic`.
- Keep this as native NetBox `extras.taggeditem`, separate from ACI plugin
  policy objects.

### Query Contract And API Budget Gates

- The existing query-contract audit now includes an optional-plugin query
  contract registry for the ACI integration.
- The audit remains generic enough that routing, peering, ACI, and future
  optional plugin maps can use the same contract summary helper.
- Ensure optional maps cannot be enabled unless their saved query IDs and
  parameter defaults are release-gate clean.
- Add a CI gate that verifies the shipped query set is published and bound in
  the validation org repository folder, and fail the gate when a shipped path
  is missing, stale, or no longer parameter-compatible. **Status: implemented
  via the `forward_validation_org_query_audit` command and CI workflow step.**
- Keep the validation-org sync check generic and repo-driven; it should compare
  the bundled query manifest against the published `/forward_netbox_validation`
  paths, then confirm the live query bindings resolve to the committed query
  shapes. **Status: implemented.**
- Include ACI NQE call counts, pages, and skipped rows in support bundles if
  the ACI path is enabled.

### Existing Optional Plugin Cleanup

- Inventory current optional routing and peering integration behavior.
- Identify which pieces already match the new registry and adapter contract.
- Move only low-risk metadata into the generic registry in `1.3.2`; avoid a broad
  routing/peering refactor unless tests show a clear behavior-preserving path.
- Add no-plugin tests for each registered optional integration.
- Add docs that describe optional-plugin support as a framework, not separate
  one-off features.

### Idempotence Coverage

- Add repeat-sync tests for any new ACI adapter.
- Add no-plugin tests proving the optional path is inert when `netbox-cisco-aci` is
  absent.
- Add stale-object/delete tests only after the create/update identity contracts
  are stable.

### Documentation

- Document the native ACI scope versus optional ACI plugin scope.
- Document what the integration does not do: it does not replace APIC as the
  canonical source for full policy, and it does not map ACI policy objects into
  approximate native NetBox objects.
- Add a short migration note for operators who already have native ACI devices
  imported as `NXOS`.

## Validation

Minimum development checks:

- `invoke harness-check`
- focused adapter tests
- focused query registry tests
- optional integration registry tests
- no-plugin startup and sync tests
- `invoke lint`
- `invoke check`

Minimum release checks:

- `invoke harness-test`
- `invoke scenario-test`
- `invoke test`
- `invoke docs`
- `invoke ci`

Live validation:

- Execute updated native platform/device queries through saved query IDs.
- If ACI plugin maps are added, execute them against a redacted or approved
  validation dataset and record only aggregate row counts and model names.
- Run a first sync and immediate repeat sync with ACI maps enabled in a local
  NetBox instance that has `netbox-cisco-aci` installed.
- Confirm Forward API/NQE usage remains bounded and does not use column filters
  for repeated same-column values.

Current sanitized validation evidence:

- NetBox `4.6.1` with `netbox-cisco-aci` installed starts cleanly and applies
  the external plugin migrations.
- The optional ACI maps seed disabled by default; only the six proven maps were
  enabled for smoke testing: fabric, pod, node, tenant, VRF, and filter.
- Live query preflight validated all six enabled maps against an approved field
  dataset. The smoke path used saved query execution with model-level queries,
  not repeated column filters or per-value NQE calls.
- First fast-bootstrap smoke completed with no ingestion issues. Aggregate
  created objects were: fabrics `7`, pods `7`, nodes `204`, tenants `163`,
  VRFs `165`, and filters `299`.
- Immediate repeat fast-bootstrap smoke completed with no ingestion issues and
  zero ACI object-change delta.
- Forward API usage stayed bounded in both runs: `15` HTTP attempts, `0`
  retries, `0` HTTP 429 failures, `12` NQE query calls/pages, and no diff calls.
- The validation org repository folder `/forward_netbox_validation` was updated
  through the Forward API with all `12` ACI query files for the `1.3.2` shape.

## Rollback

- Disable optional ACI NQE maps to return to native-only sync behavior.
- Remove or disable the ACI adapter registration if plugin-model application is
  defective.
- Since the native sync path remains unchanged, rollback should not require
  changing existing native NetBox devices, interfaces, IPAM rows, or FHRP rows.
- If plugin objects were created in a test environment, use NetBox Branching
  discard before merge or delete only rows tagged/owned by the Forward sync.
- If the generic registry causes unexpected behavior, disable the affected
  integration registration and leave existing native and routing/peering sync
  paths on their previous direct code paths.

## Decision Log

- Target `netbox-cisco-aci` first because it is the active validated plugin
  surface.
- Keep the implementation adapter-based so `pheus/netbox-aci-plugin` can be
  evaluated later without changing core native sync behavior.
- Add a generic optional-plugin registry because `forward-netbox` now needs to
  interact with multiple external NetBox plugin model surfaces. A framework
  keeps future integrations from becoming copy-pasted special cases.
- Do not force all existing routing/peering behavior through the new framework
  in one risky refactor. Start by registering capabilities and contracts, then
  migrate behavior only where tests prove parity.
- Do not fold ACI policy objects into core NetBox models. Native NetBox remains
  the right target for infrastructure inventory, but ACI policy needs an
  ACI-specific model surface.
- Do not start with contracts, filters, or static bindings. Those are valuable
  but more identity-sensitive, and they should wait until source data shape is
  proven.
- Do not add APIC collection to `forward-netbox` in this tranche. If APIC is
  needed as the canonical source, that belongs in the ACI plugin or a separate
  optional collector boundary.
