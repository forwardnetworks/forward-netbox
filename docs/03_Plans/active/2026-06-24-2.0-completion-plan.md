## 3. Remaining Items

### 3.1 Device_type PID-suffix churn fix (task #39)

**Context.** The ~10-change-per-sync churn on `dcim.device.device_type` is **not** a single-query slug mismatch (the original hypothesis `ISR4331` vs `ISR4331/K9` is the *symptom*). The real mechanism is a **dual `dcim.device` workload**: two enabled `dcim.device` builtin maps run every sync and emit a row for the *same* device with **different** `device_type`/`device_type_slug`.

Live `ForwardNQEMap` state (`forward-netbox-netbox-1`):

| name | model | enabled | weight |
|---|---|---|---|
| Forward Devices (`forward_devices.nqe`) | dcim.device | **True** | 600 |
| Forward Devices with NetBox Device Type Aliases (`forward_devices_with_netbox_aliases.nqe`) | dcim.device | **True** | 2100 |
| Forward Device Models (`forward_device_models.nqe`) | dcim.devicetype | True | — |
| Forward Device Models with NetBox Device Type Aliases | dcim.devicetype | **False** | — |

`get_maps` (`forward_netbox/utilities/sync_facade.py:84-91`) returns every `enabled=True` map; `_resolve_map_query_specs` (`forward_netbox/utilities/query_registry.py:904-913`) returns **all** enabled builtins for a `model_string` when no custom map exists — there is no "pick one variant" logic. So both `dcim.device` queries run and emit conflicting rows:
- `forward_devices.nqe` (lines 24, 30-31): `device_type = model` (raw PID), `device_type_slug = slugifyNetboxModel(model)` → `device_type='ISR4331/K9'`, slug `isr4331-slash-k9`.
- `forward_devices_with_netbox_aliases.nqe` (lines 45-62): alias-maps the PID through the NetBox Device Type Library → `device_type='ISR4331'`, slug `cisco-isr4331`.

Both target DeviceTypes already exist (pk=127 `ISR4331/K9`/`isr4331-slash-k9`; pk=141 `ISR4331`/`cisco-isr4331`). The two workloads reconcile independently — no cross-workload dedup by device name (confirmed in `apply_engine_decision.py`). The lower-weight workload sets the FK; the higher-weight workload overwrites it. Each pass the Branching ChangeDiff records the flip; the next sync flips it back → the churn. The `dcim.devicetype` side is *inconsistently paired*: its alias variant is **disabled** while the device alias variant is **enabled**, which is what left both DeviceType rows persisted and selectable.

**Why the two originally-proposed fixes do NOT work.**
- *"Strip the trailing `/K9` PID in the NQE model"*: makes `forward_devices.nqe` emit `ISR4331`, but the alias query already emits `ISR4331` and the DTL slug is `cisco-isr4331`, **not** `slugify('ISR4331')='isr4331'`. The two workloads would still disagree on slug, churning `dcim.devicetype` instead. It also risks collapsing genuinely distinct PIDs (`ISR4331/K9` vs `ISR4331-SEC/K9` both → `ISR4331`). And it edits a query body → would require an NQE org-commit.
- *"Align the device apply's device_type resolution"*: all apply paths already resolve slug-first then model (`apply_engine_bulk.py:1193-1195`, `sync_device.py:160-168`, `sync_runner_adapters.py:433-450`). They resolve correctly *per row* — the problem is two rows per device, not a resolution bug.

**The fix — Python/config-only, effective immediately, no `.nqe` body change, no NQE org-commit.** Enforce a single enabled query per `model_string` for the mutually-exclusive alias/non-alias variant pairs. Two parts:

1. **Immediate remediation (data, this environment).** Disable the redundant device workload so exactly one runs. The alias variants are the intended path, so disable `forward_devices.nqe` (weight 600) and **enable** `forward_device_models_with_netbox_aliases.nqe`, so the device and devicetype sides use the **same** alias mapping and agree on `ISR4331`/`cisco-isr4331`. (Equivalently: disable both alias variants and keep the two raw queries — either way the pair must match.) No commit, no the org.
2. **Guard (code, ship with 2.0).** Add a mutual-exclusion constraint in variant resolution. In `_resolve_map_query_specs` (`query_registry.py:904-913`): when multiple enabled builtin maps share the same `model_string` and belong to a known variant family (raw vs `*_with_netbox_aliases`), select **one** — prefer the alias variant when its companion data file `netbox_device_type_aliases` is attached, else the raw variant — instead of returning both. The variant family is the `(base_filename, alias_filename)` pairs already enumerated in `BUILTIN_OPTIONAL_QUERY_MAPS` (`query_registry.py:352-364`). Extend `signals.py:51-77` so enabling/disabling a variant for one model also flips its companion model, keeping `dcim.device` and `dcim.devicetype` variants in lockstep.

**Exact steps / commands.**

```bash

docker exec forward-netbox-netbox-1 python manage.py shell -c "
from forward_netbox.models import ForwardNQEMap

    m = ForwardNQEMap.objects.get(name=name); m.enabled = on; m.save(update_fields=['enabled'])
    print(name, '->', on)
setflag('Forward Devices', False)                                          # raw dcim.device OFF
setflag('Forward Devices with NetBox Device Type Aliases', True)           # alias dcim.device ON
setflag('Forward Device Models', False)                                    # raw dcim.devicetype OFF
setflag('Forward Device Models with NetBox Device Type Aliases', True)     # alias dcim.devicetype ON
"
# Verify exactly one enabled per model for the variant pair:
docker exec forward-netbox-netbox-1 python manage.py shell -c "
from forward_netbox.models import ForwardNQEMap
for ms in ('dcim.device','dcim.devicetype'):
    qs = ForwardNQEMap.objects.filter(model=ms, enabled=True)
    print(ms, qs.count(), [m.name for m in qs])"
# Expect: dcim.device 1 [...alias...]   dcim.devicetype 1 [...alias...]
```

Then implement the code guard in `_resolve_map_query_specs` and the `signals.py` lockstep, and add the tests below.

### A.1 Clean-DB reset (run before each clean run)
Branch schemas are `branch_<id>` (`schema_prefix='branch_'`). The ingestion-issue → ingestion → (branch, job) FK chain means CASCADE handles issues. `core_job` is **not** touched by the data truncate, so realign its sequence afterward or the harness `Job.objects.create()` (no explicit id → `nextval`) can collide.

```bash
docker exec forward-netbox-netbox-1 python manage.py shell -c "
from django.db import connection
with connection.cursor() as c:
    # 1) Drop any leftover branch schemas (named branch_<schema_id>)
    c.execute(\"SELECT schema_name FROM information_schema.schemata WHERE schema_name LIKE 'branch_%'\")
    for (s,) in c.fetchall():
        c.execute(f'DROP SCHEMA IF EXISTS \"{s}\" CASCADE')
        print('dropped schema', s)

    # 2) Forward + branching bookkeeping (ingestion_issue cascades from ingestion)
    c.execute('DELETE FROM forward_netbox_ingestion_issue')
    c.execute('DELETE FROM forward_netbox_ingestion')
    c.execute('DELETE FROM netbox_branching_appliedchange')
    c.execute('DELETE FROM netbox_branching_changediff')
    c.execute('DELETE FROM netbox_branching_branchevent')
    c.execute('DELETE FROM netbox_branching_branch')

    # 3) Truncate device-derived NetBox data (flush leaves mac/ip/devicetype behind)
    c.execute('''TRUNCATE TABLE
        dcim_macaddress, ipam_ipaddress, ipam_prefix, ipam_vlan, ipam_vrf,
        dcim_interface, dcim_inventoryitem, dcim_module, dcim_modulebay,
        dcim_cable, dcim_virtualchassis, dcim_device, dcim_devicetype,
        dcim_platform, dcim_location, extras_taggeditem, core_objectchange
        RESTART IDENTITY CASCADE''')

    # 4) Realign core_job sequence so the harness Job.create() never collides
    c.execute(\"SELECT setval('core_job_id_seq', COALESCE((SELECT MAX(id) FROM core_job), 0) + 1, false)\")
    print('reset OK')
"
```

Notes:
- `extras_taggeditem` is truncated so device-attached tags reset cleanly; `extras_tag` definitions are intentionally kept.
- `dcim_site` is kept (network-scoped, created idempotently); add it to the TRUNCATE only for a fully bare DB.
- To fully clean `core_job` too, add `core_job` to the TRUNCATE list — then step 4's `setval` resets it to 1. Leaving `core_job` intact is fine; step 4 still prevents collisions.
