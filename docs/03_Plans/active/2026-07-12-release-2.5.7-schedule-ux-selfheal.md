# Release 2.5.7 — standing-schedule UX: form management, self-healing intent, cron-friendly responses

## Goal

Close the deferrals accepted in 2.5.6's Decision Log: manage standing
schedules from the sync form and detail page, make schedules survive worker
crashes and mid-run changes via desired-state intent stored on the sync,
answer retry-blind schedulers idempotently, and stop recurring validation
from accumulating run history forever.

## Constraints

- Upgrade safety: 2.5.6 created schedules WITHOUT stored intent — reconcile
  must adopt those chains (backfill intent from the live row), never cancel
  them. Hence: explicit cancel stores `0`; an ABSENT key means "pre-intent
  install".
- A RUNNING occurrence's Job row must never be deleted (core recurrence
  re-INSERTs and re-enqueues from in-memory state — schedule resurrection);
  cancellation happens through the intent, which every occurrence re-checks.
- Responses to automation must be truthful: `already_running` only when the
  requested work is actually queued/running; the prune-during-sync refusal is
  a distinct `blocked_by_sync_run`.
- No migrations; `netbox_routing` stays optional.

## Touched Surfaces

- `utilities/sync_facade.py` — intent storage
  (`validation_schedule_interval`/`preview_schedule_interval` in
  sync.parameters), `persist_standing_schedule_interval`,
  `cancel_standing_schedule` (pending/scheduled rows only),
  `reconcile_standing_schedules` (adopt / ensure+sweep / cancel),
  `JobBlockedBySyncRun`.
- `jobs.py` — occurrence-side intent guard (cancelled → chain stops; changed
  interval → chain re-aligns; absent → backfill), end-of-sync reconcile with
  fresh parameters, isolated `_trim_validation_runs` retention.
- `forms.py` — Standing Schedules fieldset (two interval fields), parameters
  round-trip, reconcile on save.
- `api/serializers.py` + `api/views.py` — `interval: 0` cancel, 200-vs-201
  on re-posts, 202 `already_running`/`blocked_by_sync_run`, key-presence
  schedule guard on non-schedulable actions, OpenAPI response declarations.
- `utilities/model_validation.py` — allowlist + validation of the intent
  keys (non-negative int; preview ≥ 60 even on raw PATCH; absent ≠ 0).
- `views.py` + `templates/forwardsync.html` — standing-schedules rows on the
  sync detail page.
- Tests: `test_scheduled_jobs.py` (+30), `test_button_jobs.py` (202
  contract), `test_device_scope_reconciliation_audit_command.py` (skip-gated
  real-schema sweep test).

## Approach

Single hardening pass on `feat/2.5.7-schedule-ux`, adversarially reviewed by
a 24-agent workflow (3 lenses + refutation) before commit; the 19 confirmed
findings — including the upgrade-adoption blocker and the mid-run
cancel/replace resurrection mechanics (verified against core `JobRunner`,
`Job.delete`, and RQ sources in the dev container) — were fixed in the same
pass. Squash-merge via PR, tag `v2.5.7`, Trusted Publishing, GitHub release
from CI artifacts.

## Validation

- Full suite: 1117 tests OK (29 skipped); pre-commit `--all-files` green.
- Invariants pinned: intent persistence for both kinds, cancel stores 0 and
  removes rows, adoption of intent-less chains, occurrence guard
  (cancel/re-align/backfill), blocked_by_sync_run 202, interval-0-on-prune
  400, retention keeps the NEWEST N (pk-set assertion), form initial/save
  round-trip + reconcile call.
- 2.5.6 upgrade path: reconcile + occurrence guard both adopt intent-less
  chains (no silent cancellation).

## Rollback

`pip install forward-netbox==2.5.6` and restart; no migrations. Intent keys
in sync.parameters are inert on 2.5.6 (fail its allowlist only on the next
form save, which rebuilds parameters without them). Standing schedules keep
running on 2.5.6 semantics.

## Decision Log

- Cancel = stored `0`; absent key = adopt (upgrade safety). Deleting the Job
  row from the Jobs list is no longer a cancel — the intent recreates it.
- RUNNING occurrences are never deleted; the intent guard makes stray chains
  self-terminate or re-align at their next firing (at most one no-op
  firing after a mid-run cancel).
- `blocked_by_sync_run` stays 2xx (cron-green) but is distinct so callers
  can detect that the prune did not run; post-sync auto-prune covers the gap.
- Remaining accepted gaps (documented, not coded): REST `PATCH` of intent
  keys takes effect at the next reconcile point rather than immediately;
  the read-modify-write window on concurrent parameter writers is narrowed
  (fresh DB read) but not transactional.

## Bundled changes

- **Standing schedules in the UI** — sync form "Standing Schedules" section
  (recurring validation / dependency preview, minutes, blank disables) and
  schedule rows with next-run time on the sync detail page.
- **Self-healing schedules** — intent stored on the sync; recreated at the
  end of every sync run (hard-killed worker no longer silently kills the
  chain), re-checked by every occurrence (cancelled chains stop themselves,
  re-configured chains re-align, no duplicate chains from mid-run changes),
  and 2.5.6-created schedules are adopted automatically on upgrade.
- **Cancel semantics** — blank the form field or `POST {"interval": 0}`
  (`200 {"status": "cancelled"}`); deleting the sync cancels its schedules.
- **Cron-friendly API responses** — `202 already_running` replaces `409`
  for duplicate button jobs and validate; prune refused during a sync run is
  a distinct `202 blocked_by_sync_run`; idempotent schedule re-posts answer
  `200`; schedule bodies on non-schedulable actions are `400`; OpenAPI
  declares the new shapes.
- **Validation-run retention** — recurring validation keeps the newest 100
  runs per sync (`validation_run_retention` plugin setting, 0 disables).
- Hardening: preview's 60-minute floor and intent-key validation enforced on
  raw REST parameter writes; prune job data now includes
  `pruned_dangling_rows`; retention trim can never fail a successful
  validation occurrence.
- Python-only; no NQE query change — no validation-org republish needed.
