## Codex investigation verdict (read-only, gpt-5.5) — CONFIRMED

**Stale pinned `query_id` CAN silently produce missing rows with no
Issues-panel issue.** Evidence (file:line):

- Direct `query_id` **wins over** bundled raw NQE at fetch:
  `query_registry.py:804`, `:994`.
- **Direct IDs get NO repository freshness lookup; only `query_path` specs
  resolve live each sync:** `query_registry.py:864`. → a map on a direct id
  never auto-follows query changes.
- Fetch passes the saved id straight to Forward: `query_fetch_execution.py:2463`;
  client sends `queryId`(+optional `commitId`): `forward_api_impl.py:1596`.
- A valid-but-old id returning `[]` → `row_count=0` → **no workload → no apply
  → no issue** (silent): `query_fetch_execution.py:1603`.
- An INVALID id → `ForwardClientError` (400/404, `forward_api_impl.py:62/730`)
  → recorded as a model failure (`failure_count=1`,
  `query_fetch_execution.py:860/1531/1698`) but **NOT necessarily an
  Issues-panel row** unless a drift policy blocks (`validation.py:114/179`).
  So even a broken id is "transport-loud, Issues-panel-conditional" (shows in
  `model_results`/logs, not the Issues tab).
- "32 maps need Refresh" is NOT proof all 32 are stale — every direct id is
  locally classed `direct_query_id_unverified` because local Health can't
  inspect the Forward source (`query_binding_resolution.py:178`). Live drift
  distinguishes not-found vs mismatch (`:279/:359`).
- Refresh Query IDs only works if the current CVE/Vuln queries already exist
  in `/forward_netbox_validation/`; else Publish first
  (`query_binding_resolution.py:553/778/663/1080`).

## THE FIX to build (Craig approved the approach)

**Make Publish bind maps to the repo `query_path` (live, auto-resolves to
current head each sync) and CLEAR `query_id`, instead of pinning a `query_id`
snapshot.** Then there is nothing to go stale, and "Refresh Query IDs" becomes
unnecessary (Craig: "refreshing is just confusing"). Operators who want a fixed
old revision set a **commit_id** (the pin escape hatch, already supported).

### Exact current behavior to change
`forward_netbox/utilities/query_binding_resolution.py`,
`apply_explicit_nqe_map_bindings` (the PUBLISH writeback), lines **~1270-1273**:
```python
query_map.query_id = binding.query_id      # <- pins a snapshot id
query_map.query_repository = binding.query_repository
query_map.query_path = ""                  # <- clears the live path
query_map.query = ""
query_map.commit_id = binding.commit_id
```
This is the smell: publish CLEARS the live `query_path` and PINS a `query_id`
that then never re-resolves. Flip it so publish keeps/sets `query_path`
(repo-path binding, live) and clears `query_id` (unless a commit pin is
intended).

### Design decisions (confirmed with Craig)
- Publish → path-binding (live). No auto-refresh on every sync (would defeat
  intentional pins + mutate queries mid-run + add a Forward round-trip/failure
  to the sync path — all rejected).
- Fixed revision = set `commit_id` (pin). Refresh respects existing commit
  pins (`query_map.commit_id or head`), so pins survive.
- "Refresh Query IDs" concept is confusing — folding its purpose into Publish.
  DECIDE: keep the Refresh button as a legacy no-op/alias, or remove it +
  its view (`views.py:1269 ForwardSyncRefreshQueryIdsView`) + template button.
  Lean: keep the view working (harmless) but de-emphasize in UI/docs; or make
  Publish the single button. Craig's intent = simplify to Publish only.
