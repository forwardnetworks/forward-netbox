# 2.0 Legacy Teardown — single-branch is the only path (2026-06-22)

## Goal

Now that single-branch ingest (`ForwardSingleBranchExecutor`) is the proven,
scale-validated default, remove the superseded per-shard machinery and collapse
to ONE execution path. This is a breaking change → **2.0**. Aggressive scope
(operator's choice): delete the per-shard executor, resumable per-shard jobs,
10k budget sharding, and the non-branching fast-bootstrap backend; slim the
execution-ledger to single-branch-only; prune legacy management commands + UI.

## Constraints

- netbox-branching stays the engine (NetBox 4.6.3 + netbox-branching ≥1.1.0).
- Single-branch correctness + scale wins (Phases 1–4 of the bulk-merge plan) must
  not regress: full suite stays green after EVERY phase (never a broken state).
- Keep run-history/observability that the UI + merge heartbeat depend on; only
  remove the per-shard-resume bits of the ledger.
- No customer identifiers in diffs/tests/docs.

## Approach

Phased, suite-green after each phase. Map-then-delete (two read-only mapping
agents produce the exact reference checklists for the ledger and the
commands/UI). Keep the helpers single-branch reuses (`run_item_in_branch`,
`build_branch_plan`, `BranchWorkload`/`BranchPlanItem`, `create_noop_ingestion`,
`set_runtime_phase`); delete everything that exists only to serve per-shard
sharding/resume.

## Decisions + mapping findings (locked)

- **Run-history: RIP IT ALL OUT** (operator). Critical finding (ledger mapping):
  `ForwardSingleBranchExecutor` **never creates ForwardExecutionRun/Step** — runs
  exist only on the per-shard path. So on the default path the entire
  execution-ledger (~5,580 lines) + run-history UI is already inert (every call
  returns None/no-op). The single-branch merge needs only a ~4-function no-op
  shim (`touch_execution_step_progress`→False, `execution_step_for_ingestion`→None,
  `mark_ingestion_step_merged`/`mark_run_completed`→None). Delete the
  `execution_ledger*` family, the ForwardExecutionRun/Step models (+ migration to
  drop tables), and the run-history UI (tables/views/api/navigation/serializers/
  filtersets ExecutionRun/Step surfaces).
- **smoke_sync is entangled** — `tests/test_apply_engine.py` reuses
  `forward_smoke_sync.Command` build helpers for ~8 bulk-ORM tests. Before
  deleting the command, relocate those build_source/build_sync helpers into a
  test helper (or rewire the command to single-branch). NOT a clean leaf.
- **Migration(s) needed**: `ForwardExecutionRun.backend` default→SINGLE_BRANCH +
  trimmed choices + data-migrate legacy rows; `max_changes_per_branch` field
  removed/re-defaulted; drop ExecutionRun/Step tables.
- Full UI/forms/health/sync_state/runtime_guidance reference checklist captured
  by the commands/UI mapping agent (collapse every BRANCHING/FAST_BOOTSTRAP arm
  to single-branch; delete the `execution_backend`/`max_changes_per_branch`/
  `scheduler_overlap` form fields + table/serializer/filterset `backend`).

## Bundled changes (phasing)

1. **Backend collapse** — `choices` → SINGLE_BRANCH only; `models` default +
   migration; `forms` selector removed; `sync_orchestration` single executor;
   `sync_state`/`health`/`runtime_guidance` conditionals simplified.
2. **Per-shard executor + resume removal** — delete `multi_branch_executor`,
   `multi_branch`, `resumable_branching`; rewire `jobs.py` to run single-branch
   inline; strip resumable calls from `multi_branch_lifecycle`/`ingestion_merge`/
   `sync_facade`.
3. **Budget de-shard** — remove sharding from `branch_budget`; `build_branch_plan`
   yields one item per workload.
4. **Ledger slim** — remove per-shard-resume ledger code; keep run-history +
   heartbeat (from ledger mapping agent).
5. **Command/UI prune** — delete legacy commands; collapse UI/api backend surfaces.
6. **Version 2.0.0** + docs/CHANGELOG + version tables.

## Rollback

Single squashed 2.0 branch; revert the branch to restore the per-shard path
(kept in git history). Each phase is its own commit for granular revert.

## Decision Log

- **Aggressive teardown** (operator choice) — starting from scratch; no external
  users on the per-shard path besides single-writer Partner, who moves to
  single-branch.
- **Keep the ledger's run-history + heartbeat** — it is observability, not
  per-shard plumbing; only the resume-specific parts go.
- **Branching-only** — fast-bootstrap (direct-to-main, no branch) removed to make
  branching the single, enforced path.
