# Release 1.6.1 — 1.6.0 Follow-ups

## Goal

Mature the 1.6.0 features and harden the release tooling, across feature, code-
quality, and gitops lenses, in one release.

## Constraints

- GA Forward only; read-only overlays stay off the Branching sync path.
- No live Forward call on page render.
- Live-validate device-analysis changes against CustomerOrg.
- No customer data/credentials/network ids in repo/tests/docs.

## Touched Surfaces

- `forward_netbox/models.py` (+migration) — `ForwardDeviceAnalysis` → NetBoxModel
  with a `device` FK and `sync` scope.
- `forward_netbox/{tables,filtersets,navigation,urls,views}.py`,
  `forward_netbox/api/` — list view, table, filterset, REST API, nav.
- `forward_netbox/utilities/device_analysis.py`, `queries/forward_device_analysis.nqe`
  — up-interface blast-radius refinement; sync-scoped panel/query.
- `forward_netbox/template_content.py` — sync-scoped panel + render test.
- `forward_netbox/jobs.py`, `sync_orchestration` hook — optional auto-refresh.
- `forward_netbox/utilities/health.py` + a scheduled job — collection-gap
  notification.
- `templates/.../forwardsync_run_history.html` — ingestion drill-down links.
- `scripts/release.py` — robust CI-wait retry, `--with-django`, auto notes.
- `CHANGELOG.md`, `.pre-commit-config.yaml` (commitlint).
- Version bump 1.6.1 + 3 tables.

## Approach

### A. Device analysis maturation
- Promote `ForwardDeviceAnalysis` to `NetBoxModel` (changelog + REST API +
  filterset + table) with a real `device` FK (not just device_name) and keep the
  `sync` scope. Migration recreates the table.
- List view (`ObjectListView`) + table + filterset: sort by blast_radius/cve_count,
  filter by sync/reachable. Nav menu item.
- Scope the device panel to a single analysis row deterministically (prefer most
  recent per device); the model's `device` FK removes name ambiguity.
- `blast_radius` counts links only on operationally-up interfaces (active degree).
- Optional auto-refresh: a post-sync hook enqueues the refresh job when enabled.
- Panel render test via the template extension.

### B. Collection-gap notification
- A management command / schedulable job refreshes the backfilled tag and, when
  the count crosses a threshold, records a NetBox notification / log signal so
  operators are alerted without polling the health page.

### C. Run-history drill-down
- Link each run row to its `ForwardIngestion` detail (changes/issues).

### D. GitOps
- `scripts/release.py`: ret--retrying CI wait (poll `gh run view` JSON until both
  matrix jobs conclude), `--with-django` to run the container suite, and notes
  auto-derived from the release plan's Bundled changes.
- `CHANGELOG.md` generated from tags; keep-a-changelog format.
- commitlint pre-commit hook enforcing conventional commits.

## Validation

- Full `forward_netbox.tests` (Django) incl. new model/list/API/panel tests.
- Local CI mirror; NQE linter clean; harness.
- Live CustomerOrg validation of the refined device-analysis NQE.
- GitHub CI both matrices.

## Rollback

Each item is independent and revertable; the model promotion is an isolated
read-only table (migration revert drops it). No Branching-path impact.

## Decision Log

- NetBoxModel promotion (vs plain model): operators want fleet-wide triage
  (list/sort/filter) and API access; NetBoxModel gives all three for modest
  boilerplate and makes the list view native.
- `device` FK over device_name string: removes multi-sync ambiguity and gives the
  panel a direct, unambiguous row.
- Notification via a schedulable job (not on render): keeps the health page cheap
  and lets operators run the check on their cadence.
