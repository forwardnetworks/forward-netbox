# Release 1.5.6 — Resilient Duplicate Global IP Coalesce

## Goal

Stop `ipam.ipaddress` rows from failing with
`ForwardSearchError: Ambiguous coalesce lookup` when NetBox holds more than one
global (VRF-less) IP address for the same host. Seen live on CustomerOrg for reused /30
link ranges (`192.168.1.1/30`, `192.168.1.2/30` present multiple times).

## Constraints

- Do not delete or merge the duplicate IPs (pre-existing data the plugin did not
  create); only assign the synced interface to one of them.
- Deterministic selection so repeated syncs do not churn.
- Surface the duplicate to operators (aggregated warning) without breaking sync.

## Touched Surfaces

- `forward_netbox/utilities/sync_ipam.py` — `apply_ipam_ipaddress` null-VRF
  branch resolves multiple `(address__net_host, vrf__isnull)` matches itself
  instead of calling `_get_unique_or_raise` (which raises on >1).
- `forward_netbox/tests/test_ipaddress_dup_global.py` — regression test.
- `pyproject.toml`, `forward_netbox/__init__.py`, the three release tables —
  bump to 1.5.6.

## Approach

When the null-VRF host lookup returns multiple global IPs, prefer the copy
already assigned to the current interface; otherwise take the lowest-pk match.
Record a `duplicate-global-ip` aggregated skip warning. Then run the existing
create/update logic against the chosen object. The bulk path already tolerates
duplicates (it keys existing IPs and the last one wins, no error), so only the
adapter needed the fix.

## Validation

- `forward_netbox.tests.test_ipaddress_dup_global` (no exception; one global copy
  ends up on the synced interface; duplicates remain).
- `forward_netbox.tests.test_bulk_adapter_parity` (ipaddress parity unchanged).
- Full suite; local CI mirror (pre-commit run from a clean cache, twice; harness
  + harness tests; py_compile; mkdocs --strict; Django check; build).
- GitHub CI on both matrices.

## Rollback

Revert `apply_ipam_ipaddress` to `_get_unique_or_raise` and drop the test +
version bump. No schema/data impact.

## Decision Log

- Resolve-and-warn over fail-the-row: a duplicate global IP is a data condition,
  not a reason to drop the assignment; failing created noisy ingestion issues.
- Prefer the interface-assigned copy so steady-state syncs are idempotent and do
  not bounce the assignment between duplicates.
