# Release Plan: 0.8.1

## Goal

Prepare `0.8.1` as a patch release for the fast-bootstrap visibility fixes,
runtime timeout guidance, and transient Forward API retry hardening.

## Constraints

- Keep Branching as the default reviewable backend.
- Keep fast bootstrap branchless and limited to trusted initial baselines.
- Use native NetBox `ObjectChange` records for fast-bootstrap audit/statistics.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or
  private screenshots.
- Require local validation and GitHub Actions success before publishing
  artifacts.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Fast-bootstrap direct-write tracking and ingestion changes view
- Forward API retry behavior
- Runtime timeout guidance
- Tests, release docs, and operator guidance

## Approach

1. Store a direct-write request id for fast-bootstrap ingestions.
2. Run fast-bootstrap writes inside native NetBox request/event tracking.
3. Populate branchless ingestion counters and changes from NetBox
   `ObjectChange` rows.
4. Warn operators when NetBox worker timeout is undersized for Forward API or
   large Branching workloads.
5. Retry transient Forward API HTTP timeout/gateway responses using the existing
   source retry setting.
6. Update release metadata and docs for `0.8.1`.
7. Run the full repo gate, commit, push, and verify GitHub Actions before
   publishing release artifacts.

## Validation

- Targeted fast-bootstrap direct-change tests
- Targeted Forward API transient HTTP retry tests
- Runtime timeout guidance tests
- `invoke harness-check`
- `invoke harness-test`
- `invoke docs`
- `invoke lint`
- `invoke check`
- `invoke test`
- `invoke ci`
- GitHub Actions success on `main` and `v0.8.1`

## Rollback

If local or GitHub validation fails, keep `0.8.0` as the latest published release
and revert the `0.8.1` metadata/docs commit. If a release is published and a
blocker is found, publish a newer patch release rather than replacing immutable
artifacts.

## Decision Log

- Chosen: native NetBox `ObjectChange` tracking for fast bootstrap because the
  backend is branchless and should not synthesize Branching diffs.
- Chosen: retry transient HTTP responses alongside transport timeouts because
  gateway timeouts at sync start are recoverable and already bounded by the
  source retry setting.
- Rejected: making fast bootstrap the default because Branching remains the
  reviewable steady-state workflow.
