# Forward NetBox 1.3.1 Prefix Accounting and Parameterized Fetch Release

## Goal

Ship `forward-netbox` `1.3.1` as a patch release on top of the `1.3.0`
parameterized NQE tranche.

The release fixes repeat prefix syncs where unchanged `ipam.prefix` rows could
still appear as updates in ingestion statistics and branch diffs, and removes
the legacy sync execution path that used Forward NQE column filters for shard
fetches.

## Trigger

A production repeat sync showed every prefix shard reporting thousands of
updates even after a prior sync had already imported the same rows.

The `1.3.0` query path was already using query-side
`forward_netbox_shard_keys` parameters for the default shard contracts. The
remaining issues were local NetBox row matching/apply accounting and stale
runtime/test support for the older column-filter shard path.

## Constraints

- New sync execution must not call Forward NQE with `column_filters`.
- Parameterized shard execution must remain query-side through
  `forward_netbox_shard_keys`.
- Full-model fetch remains valid only for models whose shard contract is
  explicitly `model`; it must not be a compatibility fallback after a
  parameterized shard failure.
- Historical execution-step metadata fields can remain for old records and
  migrations, but generated/current sync paths must not select
  `nqe_column_filter`.
- Low-level Forward API support for column filters can remain for direct API
  compatibility and non-sync tooling, but NetBox sync orchestration must not use
  it.
- Do not introduce customer identifiers, network IDs, snapshots, credentials,
  or live tenant data into release artifacts.

## Approach

1. Make `coalesce_update_or_create` and `coalesce_upsert` optionally return a
   changed/no-change signal without changing existing callers by default.
2. Use the change-aware path for `ipam.prefix` so exact repeat rows return
   `"unchanged"`.
3. Compare NetBox `Prefix.prefix` values against incoming strings by canonical
   string form when direct equality fails.
4. Make dependency VRF creation non-mutating for prefix, IP address, routing
   dependency, and FHRP dependency imports while keeping explicit `ipam.vrf`
   import authoritative.
5. Remove live sync execution support for `nqe_column_filter`, including
   partition retry, sibling prefetch, and shard-failure full-model fallback.
6. Update auxiliary tooling so generated/current examples use
   `nqe_parameters`.
7. Keep local shard safety filtering after parameterized fetches.

## Scope

- Preserve the `1.3.0` parameterized NQE execution path.
- Remove the live `nqe_column_filter` sync execution path.
- Fail fast if stale branch state tries to run a column-filter shard.
- Treat unchanged prefix rows as unchanged instead of applied.
- Avoid dependency VRF rewrites from prefix, IP address, and FHRP imports.
- Keep direct VRF imports authoritative when the `ipam.vrf` map itself runs.
- Add focused regression coverage for repeated prefix-only syncs.
- Update release metadata and docs to `1.3.1`.

## Decision Log

- `coalesce_update_or_create` and `coalesce_upsert` now support an optional
  `return_change` result while preserving the existing two-value return for
  existing callers.
- `apply_ipam_prefix` uses the change-aware return and emits `"unchanged"` for
  exact no-op rows.
- Prefix comparison handles NetBox `Prefix.prefix` objects and string input by
  comparing their canonical string form when direct equality fails.
- Dependency VRF ensures can create missing VRFs without mutating existing VRF
  metadata. Prefix, IP address, routing dependency, and FHRP dependency paths
  use this non-mutating mode.
- Sync statistics now track an `unchanged` bucket so no-op rows can be surfaced
  separately from applied and skipped rows.
- `nqe_column_filter` is no longer an executable sync fetch mode; stale branch
  state using it raises a clear `ForwardQueryError`.
- Parameterized shard failures no longer widen into full-model fetches. They
  fail instead, which keeps Forward API/NQE usage bounded and visible.
- The low-level Forward API client keeps column-filter support because it is a
  direct API primitive and has separate API compatibility tests.
