# Forward NetBox 1.3+ Horizon

## Goal

Map the next meaningful improvements after the 1.3 parameterized-shard-fetch
tranche. The current release line already reduced avoidable Forward SaaS
pressure by moving the prefix path to query-side parameters and by updating the
live org repository query definitions. The next work should keep that direction:
fewer redundant Forward calls, better execution observability, stronger
recovery, and more model coverage without turning the plugin into a separate
sync system.

## Current State

The shipped 1.3 line has already established the main behavioral shift:

- built-in prefix queries accept `forward_netbox_shard_keys`
- the live org repository has been updated to the parameterized prefix shape
- sync execution still uses the synchronous Forward `/api/nqe` path
- the plugin already carries per-run query metadata, fetch metadata, and
  fallback reasons in its local execution ledger and health surfaces

That means the next gains are mostly about removing the remaining waste around
the query path and making the operator experience clearer when a run slows down
or falls back.

## Constraints

- Keep the plugin compatible with the current NetBox 4.5.x and 4.6.x release
  line.
- Preserve branch-native sync semantics and avoid introducing a new default
  transport.
- Keep query-side narrowing explicit and data-safe; do not widen row scope to
  save requests.
- Avoid customer-specific identifiers, snapshots, or other live tenant data in
  repo artifacts.
- Treat compatibility fallbacks as temporary migration aids only, not as a
  permanent execution path.

## Touched Surfaces

- `forward_netbox/queries/` for built-in NQE maps and query parameter shapes.
- `forward_netbox/utilities/query_registry.py` and
  `forward_netbox/utilities/query_fetch_execution.py` for query identity,
  lookup, and resolution caching.
- `forward_netbox/utilities/execution_telemetry.py`,
  `forward_netbox/utilities/health_summary_blocks.py`, and
  `forward_netbox/utilities/ingestion_presentation.py` for run visibility.
- `forward_netbox/utilities/execution_ledger_serialization.py` for support
  bundle evidence.
- `forward_netbox/tests/` for query registry, sync, health, export, and summary
  coverage.
- `docs/02_Reference/built-in-nqe-maps.md` and the live validation-folder
  query repository for operator-facing map documentation.

## Approach

1. Expand parameterized built-in coverage where a query can safely narrow
   itself.
2. Keep only the minimal shard-safety filters needed for data correctness.
3. Deduplicate query-path resolution and repeated Forward lookups within a run.
4. Surface query identity, path resolution, and fallback reasons in health and
   support surfaces.
5. Harden the query/diff contract with tests so regressions are caught when
   maps are cloned, reseeded, or exported.
6. Consider async NQE only as an optional later transport when the synchronous
   path is no longer adequate.

## Near-Term Horizon

### 1. Expand parameterized coverage beyond the prefix path

The biggest remaining API/NQE win is not a new transport. It is widening the set
of built-in maps that can use query-side parameters instead of column filters or
model-wide fetches.

What this likely means:

- identify the remaining built-in maps that still rely on broad column filters
- prefer explicit shard keys or other stable parameter families where the query
  can safely narrow itself
- keep local shard safety filters even when the query is parameterized
- preserve row shape and branch boundaries exactly as they are now

Why this matters:

- every map converted to query-side parameters reduces repeated backend work
- the plugin stays simpler if the query itself owns the narrowing logic
- the release surface gets easier to reason about because fetch contracts are
  explicit in the query source

This is the most direct continuation of the 1.3 work.

### 2. Reduce duplicate Forward calls inside a sync

The current client still paginates and retries synchronously. That is fine when
queries are fast, but it still leaves room for repeated identical calls during
one sync.

Good candidates:

- deduplicate identical fetches when a model/shard would otherwise hit the same
  query with the same parameters more than once
- cache stable repository lookups during one run so query resolution does not
  do repeated work
- continue collapsing obvious retry storms and identical page refetches into a
  single bounded path

Why this matters:

- it lowers request volume without changing the NetBox result
- it gives a cleaner answer to “why did the API count go up?”
- it tends to help both SaaS and local validation runs

This is useful even if the queries remain synchronous.

### 3. Add better execution visibility at the operator boundary

The repo already records a lot of runtime state, but the next step is making that
state more directly useful when a sync misbehaves.

Useful additions:

- a clearer summary of which models used full query execution, diff execution,
  or fallback
- per-model query identity, commit ID, parameters, and page-size evidence in the
  health or support surfaces
- a compact “top API/NQE consumers” view for one run
- an explicit “why did this model stop using the optimized path?” reason

Why this matters:

- operators can tell whether they are looking at a query-shape problem, a data
  problem, or a runtime problem
- support bundles become more actionable
- regression triage gets faster because the evidence is already in the product

### 4. Keep hardening the query and diff contract

The current work already leans on `query_id`, `query_path`, and `nqe-diffs`
where appropriate. The next improvement is making that contract more explicit
and harder to regress.

Useful additions:

- stricter tests around query identity preservation when maps are cloned or
  reseeded
- better verification that diff-eligible models continue using diff execution
  when a baseline exists
- tighter checks that compatibility fallbacks do not reappear in the query path
- more tests for parameter binding on saved org queries

Why this matters:

- it keeps the optimization effort from silently drifting back to raw query
  execution
- it protects the current 1.3 behavior as the codebase continues to grow

## Mid-Term Horizon

### 5. Prepare an async NQE transport as an optional execution path

Forward 26.6 is introducing an asynchronous NQE query-execution API. Based on
the current plugin shape, that is not the primary bottleneck right now, because
the current queries typically return quickly. But it is still worth planning
for.

Where it helps:

- staging or bulk validation workflows where a query can run long enough to
  justify non-blocking submission
- retry-heavy paths where holding a worker open is wasteful
- future automation that wants to submit work and poll later instead of waiting
  in one request

Where it does not help much:

- reducing total query volume
- improving query shape
- eliminating row-shape validation or shard planning work

The right implementation shape is likely an optional transport abstraction, not
a default behavior change. The sync logic should keep the same semantic contract
and only change how the result is fetched.

### 6. Expand model coverage where the Forward data model already supports it

Once the current query families are stable, the next gain is usually not
inventing new machinery. It is adding more model coverage using the same
contract.

Good targets are models that:

- already have stable Forward data
- have a clear row identity
- do not require a brittle fallback shape
- can benefit from parameterized shard fetches or direct diff execution

This is the cleanest path to broader coverage because it reuses the same query,
validation, and branching patterns.

### 7. Improve run recovery and operator guidance

The ledger and health surfaces have already moved in the right direction. The
next step is making recovery recommendations more explicit when a run is stuck
or partially complete.

Good additions:

- clearer recommendations for stale worker state versus handled timeout state
- easier identification of the exact model or shard that needs attention
- support-bundle exports that show the last known execution mode and fallback
  reason without requiring log spelunking

Why this matters:

- the plugin becomes easier to operate at larger scale
- support can distinguish “slow but healthy” from “actually stuck” faster

## Longer-Horizon Ideas

These are useful, but they should stay behind the near-term work above:

- more aggressive NQE-side coalescing where a single query can legally carry
  multiple values instead of many separate calls
- model-specific parameter families beyond the obvious prefix path
- more query-plan introspection surfaced to the user before a sync starts
- richer per-model capacity hints so the operator can see what will be
  expensive before they hit execute
- more targeted API/NQE pacing recommendations for large but valid syncs

## Suggested Priority Order

1. Expand parameterized query coverage where the query can safely narrow itself.
2. Remove duplicate or redundant Forward calls within a sync.
3. Improve the operator-facing execution and fallback summaries.
4. Tighten the diff/query identity contract with tests.
5. Add async NQE as an optional transport when there is a real staging or
   retry-management reason to do so.
6. Grow model coverage only where the Forward data model and row identity are
   already clean.

## Decision Log

- Prefer query-side narrowing over more column-filter fan-out.
- Prefer synchronous execution for the current fast-return paths.
- Treat async NQE as an optional transport, not a new default workflow.
- Keep NetBox-native row application and Branching semantics unchanged.
- Keep compatibility shims out of new work unless they are needed for upgrade
  safety.

## Validation Ideas

Any follow-up slice should prove at least one of these:

- fewer Forward HTTP calls for the same input
- fewer duplicate query executions in one sync
- faster or clearer failure diagnosis
- stronger diff eligibility without changing row output
- no regression in NetBox object shape, branch behavior, or support exports
