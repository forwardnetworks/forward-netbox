# Forward NetBox 1.2.3 Change Explainability And API/NQE Coalescing

## Goal

Make staged NetBox changes explainable without extra Forward API or NQE calls,
and continue reducing avoidable Forward NQE/API pressure during shard-scoped
fetches. An operator should be able to export a sync or ingestion support bundle
and see which local NetBox models and fields are driving staged updates.

## Constraints

- Do not add Forward API, NQE query, or NQE diff calls.
- Do not include raw customer row values in support bundles.
- Preserve NetBox Branching as the source for staged review data.
- Keep summaries bounded so a large branch export does not create an
  unbounded support payload.
- Keep `last_updated` and similar noisy bookkeeping fields out of field-change
  explanations.
- Coalesce compatible sibling shard fetches only when the same query, model,
  fetch mode, key family, and EQUALS_ANY column are in play.
- Preserve branch shard boundaries; prefetch may reduce Forward calls but must
  not increase the number of NetBox changes staged in the current branch.

## Approach

Add a local Branching `ChangeDiff` summary helper that aggregates:

- staged action counts
- staged model counts
- top changed fields across update diffs
- top changed fields per model
- whether the scan was truncated

Expose that helper in:

- ingestion log exports
- sync support bundles for the latest ingestion
- execution-run support bundles through each step's ingestion detail
- the ingestion detail page, with HTMX refresh alongside progress/statistics

The summary intentionally records field names and counts only. It does not
export pre-change values, post-change values, object names, query rows, or
customer-specific examples.

Add bounded sibling-shard prefetch for `nqe_column_filter` fetches:

- when the current execution step has compatible later stage steps in the same
  run, combine their single-column `EQUALS_ANY` values into one Forward request
- filter the returned rows back down to the current shard before staging
- save the extra sibling rows into the existing run-local fetch artifact cache
- let later sibling steps hit the local artifact instead of calling Forward
- keep per-step support metadata scoped to the current shard, even when the
  actual request used a wider prefetch filter

## Rollback

Remove the helper and the new support-bundle fields. Individual NetBox
`ChangeDiff` review remains available through the existing ingestion Changes tab.
Remove the ingestion-page partial include if only the UI surface needs rollback.
Disable sibling prefetch by setting `shard_fetch_prefetch_steps` to `1`, or
revert the `query_fetch_execution.py` prefetch helper and regression test.

## Decision Log

- Chose local Branching diff aggregation instead of Forward diff diagnostics
  because the user-facing question is why NetBox objects are staged for update,
  and Branching already has the authoritative local pre/post data.
- Chose field names/counts instead of raw values to keep bundles safe for
  customer redaction and support sharing.
- Chose a bounded scan with truncation metadata rather than an unbounded export
  because large Branching shards can have thousands of changes.
- Chose sibling-shard prefetch instead of increasing branch shard size because
  branch budgets protect NetBox/Branching review cost while the Forward request
  can safely retrieve a bounded superset and split it locally.
