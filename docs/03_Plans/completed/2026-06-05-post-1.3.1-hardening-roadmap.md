# Post-1.3.1 Hardening Roadmap

## Goal

Make the next tranche after `1.3.1` release-focused rather than feature-sprawl
focused. The operator-visible outcome is a sync integration that can prove
bounded Forward API/NQE usage, repeat-sync idempotence, shipped query contract
coverage, and upgrade safety before each release.

## Constraints

- Do not hold the already-published `1.3.1` release for this work.
- Keep NetBox-native and Branching-native behavior as the correctness baseline.
- Keep NQE as the row-shaping source of truth; Python must not fork query
  semantics.
- Do not reintroduce sync-time NQE column-filter shard fetches.
- Do not persist or commit customer identifiers, network IDs, snapshot IDs,
  credentials, screenshots, or raw live rows.
- Prefer release-gate evidence over one-off manual proof.

## Approach

1. Add a shipped-query contract summary that joins architecture fetch contracts
   to built-in NQE map definitions.
2. Treat every model with `fetch_mode=nqe_parameters` as requiring shipped NQE
   maps that:
   - declare `forward_netbox_shard_keys`
   - seed a default empty `forward_netbox_shard_keys` parameter
   - include an empty-list guard so the query remains runnable from the UI with
     no parameters selected
   - include at least one positive shard-membership predicate so parameterized
     execution actually constrains the NQE work
3. Include that summary in `forward_architecture_audit` and make
   `--fail-on-gap` fail on query-contract gaps.
4. Keep expanding repeat-sync idempotence coverage model by model after the
   query-contract gate lands. Priority order:
   - `dcim.device`
   - `dcim.interface`
   - `ipam.ipaddress`
   - `ipam.vrf`
   - `ipam.vlan`
   - `ipam.fhrpgroup`
   - routing plugin models
   - delete-only and mixed apply/delete shards

   Current proof points now include explicit repeat-sync no-op coverage for
   `dcim.device`, `dcim.interface`, `dcim.virtualchassis`, `dcim.macaddress`,
   `dcim.platform`, `dcim.manufacturer`, `dcim.devicerole`, `dcim.devicetype`,
   `extras.taggeditem`, `ipam.ipaddress`, `ipam.vrf`, `ipam.vlan`,
   `ipam.fhrpgroup`, cable, inventory item, module, ACI helpers, routing
   helpers, and peering session helpers.

  Device-type resolution now caches the combined manufacturer+slug lookup,
  platform and module-type resolution reuse cached manufacturer/platform and
  manufacturer/model lookups on repeat calls, routing interface alias
  resolution reuses the first successful lookup on repeat calls, VRF-scoped
  host-IP lookups now cache directly, OSPF instance lookups now cache both the
  device/process and device/VRF/process shapes, BGP router lookups now reuse
  the cached router identity on repeat calls, BGP scope lookups now reuse the
  cached scope identity on repeat calls, ASN lookups now backfill the positive
  cache on existing rows, and BGP address-family lookups now reuse the cached
  scope/address-family identity after the first resolution, which removes
  repeat-call lookup chatter from those hot paths.
- `dcim.site` now reuses its positive lookup cache after the first
  resolution, which keeps VLAN site resolution from re-querying the site table
  on repeat syncs.
- Global `ipam.prefix` lookups now participate in the cache as well, so the
  no-VRF prefix path no longer re-queries the prefix table after the first
  successful resolution.
- `netbox_peering_manager.relationship` now reuses its cached slug/name lookup
  on repeat resolution, which keeps the peering-manager relationship helper
  from paying the same query twice.
- Cable pair lookups now reuse the resolved cable on repeat syncs, so the
  cable helper no longer refreshes both terminations from the database when the
  same pair is checked again in one runner.
- BGP scope delete resolution now uses the shared unique cache for both router
  and scope lookups, so peer delete paths no longer repeat the router/scope
  query once the runner has resolved the pair.
- Interface access-VLAN resolution now uses the shared `ipam.vlan` unique cache
  so repeat interface syncs no longer re-query the site/VID pair for the same
  untagged VLAN.
- Device-type delete resolution now reuses the shared manufacturer cache, so
  the delete helper does not re-query the manufacturer table on repeated
  lookups for the same slug/name pair.
- Cable pair lookup now uses the in-memory cable IDs first and only falls
  back to a refresh when the runner does not already know the pair, which
  reduces the initial cable lookup from two refreshes to one query and keeps
  the repeated lookup cached.
- FHRP delete resolution now reuses the shared IPAddress cache for the VIP
  lookup, so the last-assignment delete path does not re-query the address row
  after the runner has already resolved it once.
- Device lookup by name now uses the strict cached getter under the hood, so
  repeated optional device lookups no longer pay a second database hit for the
  same missing name.
- Interface lookup now goes through the shared unique cache, which keeps
  repeated positive lookups and repeated misses on the same device/name pair
  from re-querying the database.
- Module bay lookup now goes through the shared unique cache, which keeps both
  repeated positive lookups and repeated misses on the same device/name pair
  from re-querying the database.
- OSPF interface alias candidates now prime the canonical interface cache up
  front, so shorthand routing-interface names no longer need a first-pass DB
  probe before the canonical interface identity is available to the sync
  runner.
- BGP peer rows now prime ASN, router, and scope identities in batch, so the
  repeated peer path can reuse the same routing identities without paying a
  per-row resolver query for the same device/asn/scope chain.
- OSPF instance rows now prime the area and instance identities in batch, so
  repeat routing state for the same device/process/area chain can reuse the
  same cached identities instead of re-reading them on every row.
- The query fetcher now caches resolved run-step ingestion IDs by run and
  target index, so repeated context resolution during the same fetch cycle does
  not keep querying the stage-step table.
- The query fetcher now caches the fully resolved context by its effective
  input signature, so repeated fetches in the same run can reuse the already
  resolved snapshot and context artifacts instead of rebuilding them and
  re-running the same lookup chain.
- The Forward client now caches snapshot lists, latest-processed snapshot
  reads, and snapshot metrics in-process, so the repeated read-only metadata
  calls that feed sync planning, health summaries, and validation do not keep
  hitting the same Forward endpoints inside one client lifetime.
- The Forward client now caches committed NQE query lookups by
  repository/path/commit, so repeated query-reference resolution can reuse the
  same Forward response instead of re-fetching the same repository metadata.
- The Forward client now caches org and repository NQE query list reads by
  directory, so repeated validation and binding checks can reuse the same
  Forward query catalog response inside one client lifetime.
- The Forward client now caches NQE query history by query ID and invalidates
  that query-related read cache on NQE mutations, so support and binding
  checks can reuse the same history response without risking stale reads after
  an update.
- The Forward client now caches network list reads and org head-commit reads,
  so the UI and NQE publishing path can reuse those deterministic metadata
  results inside one client lifetime.
- The Forward client now caches empty query-list and empty query-history reads
  too, so the "nothing changed" path no longer keeps re-hitting Forward just to
  rediscover an empty result.
- The Forward client now uses a short-lived shared read cache for deterministic
  metadata reads, so separate client instances in the same process can reuse
  network, snapshot, query catalog, and head-commit responses instead of
  repeating the same API calls.
- The shared query-read cache bumps a generation counter on NQE mutations, so
  cross-client reuse stays bounded to the current query state and does not
  leak stale catalogs or head-commit reads across publishes.
- The query-binding drift helper now uses indexed repository-query lookups for
  direct query IDs, so repeated repository catalog scans no longer walk the
  full query list for each map.
- The Forward API usage summary now reports aggregate read-cache hits, misses,
  and hit rate, so the same release and support surfaces can prove whether the
  cached Forward metadata paths are actually suppressing repeat reads.
- The support-bundle and health summary API-usage evidence now carries the
  same read-cache counters, so the cache telemetry survives export instead of
  stopping at the client object.
- The bulk tree ORM applier now caches lookup results across rows for
  `dcim.devicerole` and `dcim.platform`, and now batch-prefetches existing
  rows up front so the bulk payload no longer pays a per-row ORM lookup for
  repeated tree identities.
- Cable pair lookup now uses the shared unique cache for the cable row itself,
  so repeated cable checks reuse the same cable identity instead of re-querying
  the table by primary key.
- Cable pair lookup no longer refreshes both interface rows before checking the
  cable identity, which removes the extra round trip when the runner already
  has the terminations loaded in memory.
5. Add a release-scale fixture path only after the contract/idempotence gates
   are stable. The fixture must be redacted and synthetic enough to live in the
   repo.

## Future Improvements

### API Usage Telemetry Loop

Expose a first-class API/NQE budget report in the UI and support bundle:

- Forward HTTP attempts
- retry count
- 429 count
- NQE query calls
- NQE diff calls
- NQE pages
- throttle sleep seconds
- per-model and per-step fetch mode
- per-step query parameters, excluding raw data

This turns "we think we reduced API pressure" into an operator-visible release
signal.

### Idempotence Regression Matrix

Make repeat-sync no-op behavior a standing release gate. The minimum matrix is:

- fresh sync creates expected rows
- immediate repeat sync has zero unexpected updates
- dependency-only rows do not mutate authoritative models
- delete statistics are counted
- mixed shards keep create/update/delete/unchanged counts separate

### Shipped Query Contract Validator

Every built-in query map should be contract-checked against the architecture
fetch contract. This protects against future regressions where a model is marked
parameterized but the shipped query cannot actually run parameterized or cannot
run unfiltered in the Forward UI.

### Upgrade And Config Compatibility

Add upgrade tests for existing `1.0`, `1.2`, and `1.3` sync configurations:

- stale execution records
- old query maps
- direct query IDs
- repository-path query maps
- old saved parameter defaults

The expected result is greenfield `1.3.1+` behavior after upgrade without
silently reviving legacy column-filter fetches.

### Operator Remediation

When sync configuration is stale, emit exact remediation:

- query map name
- model
- execution mode
- missing parameter
- expected query path or query ID state
- whether the issue blocks bounded Forward API/NQE behavior

### NetBox-Side Performance

Before faster apply engines, reduce redundant NetBox ORM work:

- dependency lookup batching
- per-sync object cache priming
- fewer per-row ORM checks where adapter parity is already proven
- clearer per-model write/read telemetry

## Rollback

Rollback is a normal revert of the hardening commit. Since the first tranche is
read-only reporting plus tests, no data migration or runtime cleanup is needed.

## Decision Log

- Start with query-contract reporting because it is low-risk, catches a real
  class of regressions, and strengthens release gates without changing sync
  mutation behavior.
- Do not start with a large replay fixture; redaction and fixture-shape work is
  valuable but should not block the immediate release-gate improvement.
- Do not add TurboBulk or replica-cache work in this tranche.
