# Release 2.5.6 — operator-job automation: REST actions, standing schedules, prune dangler sweep

## Goal

Ship the automation tranche (developed as "2.6", released as 2.5.6 by
maintainer decision): REST parity + overlap guardrails for the four operator
button jobs, JobRunner standing schedules for validation and dependency
preview, the netbox_routing dangling-row sweep after prune, and the
hardening pass from the post-tranche 44-agent audit.

## Constraints

- Job-name couplings stay byte-identical for immediate runs (drift report /
  preview GET `icontains "dependency preview"`, webhook `- adhoc`/`-
  scheduled`, auto-prune `" (auto)"`).
- Status-only "active job?" queries must not treat permanently-SCHEDULED
  standing-schedule rows as sync activity (the audit blocker).
- `netbox_routing` remains optional; the sweep no-ops without it.
- Standing schedules are API-only this release (UI parity deferred —
  Decision Log).

See the tranche plan

## Approach

Seven commits on `feat/2.6.0-automation-tranche`: four tranche chunks, the
enqueue_once idempotency fix, docs, and the audit hardening commit

## Validation

- Full suite 1097 tests OK (28 skipped); pre-commit green.
- Live dev-stack smoke: enqueue_once idempotency (same params → same Job pk,
  interval change → single replaced row).
- 44-agent audit (6 review lenses, adversarial verify): 37 confirmed
  findings; blocker + should-fixes fixed in this release, deferrals logged
  below.
- CI: 2× "Validate NetBox v4.6.4" on the PR before merge.

## Rollback

`pip install forward-netbox==2.5.5` and restart; no migrations in this
release. Standing schedules created on 2.5.6 remain as scheduled Job rows —
delete them from the Jobs list after downgrading (2.5.5 has no JobRunner
classes for them; an occurrence firing after downgrade errors harmlessly).

## Decision Log

- Released as 2.5.6 instead of 2.6.0 (maintainer call).
- Immediate runs keep per-sync job names; standing schedules use fixed
  JobRunner names; all overlap guards check both shapes under core's
  job-schedules advisory lock.
- `schedule_at` requires `interval` (one-shot delayed runs would silently
  replace the standing schedule's enqueue_once slot) and must be future;
  dependency preview enforces a 60-minute interval floor.
- Deferred, accepted for a later release: UI (form) management of standing
  schedules; 202-idempotent semantics for prune under dumb crons (409 is
  documented as benign); ForwardValidationRun retention/cleanup command;
  schedule re-registration after a hard-killed worker (documented caveat);
  200-vs-201 on idempotent schedule re-posts; skip-gated real-schema test
  for the routing sweep once netbox_routing supports the 4.6 baseline.

## Bundled changes

- **Operator-job REST actions** — `POST /api/plugins/forward/sync/<id>/`
  `dependency-preview|prune-orphans|tag-delete-eligible-ipam|create-module-bays/`:
  the button's permission, `201` + job on success, `409` when an equivalent
  job is already queued/running. Duplicates never stack across HTML buttons,
  API calls, scheduled occurrences, and the post-sync ` (auto)` prune —
  one shared overlap guard under an advisory lock.
- **Standing schedules** for `validate` and `dependency-preview`:
  `{"interval": minutes}` (optional future `schedule_at`) keeps exactly one
  recurring schedule per sync; same-parameter re-posts are no-ops, changed
  parameters replace the schedule, cancel by deleting the scheduled job.
  Preview enforces a 60-minute interval floor (it is a full live dry-run —
  schedule daily or less). Snapshot catch-up and the sync self-reschedule
  are name-scoped so schedules never suppress them; deleting a sync cancels
  its scheduled jobs.
- **Duplicate Validate guard** — repeated Validate clicks/POSTs return
  `409`/a warning instead of stacking concurrent validation runs.
- **Prune dangler sweep** — after pruning devices, dangling netbox-routing
  rows (BGP routers/scopes/address-families/peers/settings whose generic
  references pointed at the pruned devices) are deleted in dependency order
  and reported as `pruned_dangling_rows` in the job data.
- Operations Guide: new "Automating operator jobs" section (endpoints,
  permissions incl. the API token's `add_forwardsync` requirement, schedule
  semantics and caveats, cancellation, retention notes).
- Python-only; no NQE query change — no ADP republish needed.
