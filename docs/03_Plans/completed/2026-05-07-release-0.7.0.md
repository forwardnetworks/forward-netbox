# Release Plan: 0.7.0

## Goal

Publish `0.7.0` with the 0.7 sync boundary cleanup, shard heartbeat visibility, and release-doc updates while keeping the release aligned with the validated NetBox 4.5.9 runtime.

## Constraints

- Keep the release line on NetBox `4.5.x`.
- Preserve the NetBox-native multi-branch workflow and branch-budget defaults.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or private screenshots.
- Require the repo release gate before tagging or publishing artifacts.
- Publish immutable GitHub and PyPI artifacts from the tagged commit.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Release compatibility and version-history docs
- GitHub release tag and notes
- PyPI wheel and source archive

## Approach

1. Bump package and plugin version metadata to `0.7.0`.
2. Update install examples and release compatibility tables.
3. Run the repo release gate and package build checks.
4. Commit the release metadata/docs change.
5. Tag, push, create the GitHub release, and upload the PyPI artifacts.

## Validation

- `invoke ci`
- `python -m build`
- `python -m twine check dist/*`
- GitHub Actions success on `main` and the release tag

## Rollback

If validation fails before publishing, revert the release metadata/docs commit and keep the previous release as latest. If publication succeeds but a release-blocking issue is found, publish a newer patch release.

## Decision Log

- Chosen: `0.7.0` major release because the 0.7 refactor is now the new default operating shape and the docs need to match the current release line.
- Rejected: publishing without a plan file because the repo gate requires a plan artifact for high-risk release changes.
