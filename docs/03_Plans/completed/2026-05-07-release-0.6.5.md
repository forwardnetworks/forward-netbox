# Release Plan: 0.6.5

## Goal

Publish `0.6.5` with the audited validation force-allow override path and the routing evidence/skip-resolution tranche, while keeping the release aligned with the validated `4.5.9` NetBox runtime and the current beta boundaries.

## Constraints

- Keep the current release line on NetBox `4.5.x`.
- Preserve the beta labels for optional routing/peering and native module import.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or private screenshots.
- Require local `invoke ci`, GitHub Actions success on `main`, and GitHub Actions success on the release tag before publishing artifacts.
- Publish immutable GitHub and PyPI artifacts from the tagged commit.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Release compatibility and version-history docs
- Validation override UI/API/model work
- Routing evidence enrichment and skip-resolution docs
- GitHub release and PyPI artifacts

## Approach

1. Bump package and plugin version metadata to `0.6.5`.
2. Update release compatibility tables and install examples.
3. Call out the audited validation override and routing evidence enrichment in release notes.
4. Re-run the repo release gate and sensitive-content scan.
5. Commit, tag, push, verify GitHub CI on both branch and tag, then publish GitHub Release and PyPI artifacts.

## Validation

- Local `invoke ci` passed after the `0.6.5` version bump, including harness checks, sensitive-content pre-commit hook, Docker build/startup checks, Django checks, plugin tests, Playwright UI harness, docs build, and package build.
- `python -m twine check dist/forward_netbox-0.6.5*` passed for the locally built wheel and source archive.
- Pending GitHub Actions verification on `main` and `v0.6.5`.
- Pending PyPI upload verification.

## Rollback

If validation fails before publishing, revert the release metadata/docs commit and keep `0.6.4` as latest. If publication succeeds but a release-blocking issue is found, publish a newer patch release because GitHub release assets and PyPI files should remain immutable.

## Decision Log

- Chosen: `0.6.5` patch release because the new work is feature-complete, beta-scoped, and does not expand the default inventory surface.
- Rejected: skipping the release notes update because the repository’s release playbook expects the public docs to match the tagged artifact.
