# Release Plan: 0.9.1

## Goal

Publish `0.9.1` with the release-doc refresh and the final cleanup sweep on the architecture tranche, while keeping the release aligned with the validated NetBox-native branch workflow.

## Constraints

- Keep the release line on the current NetBox `4.5/4.6` shared branch behavior.
- Preserve the NetBox-native sync and branch execution path.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or private screenshots.
- Require the repo release gate before tagging or publishing artifacts.
- Publish immutable GitHub and PyPI artifacts from the tagged commit.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Release compatibility and user-guide docs
- Architecture cleanup in `forward_netbox/utilities/*` and related tests
- GitHub release tag and notes
- PyPI wheel and source archive

## Approach

1. Bump package and plugin version metadata to `0.9.1`.
2. Update install examples, release compatibility tables, and operator notes.
3. Run the repo release gate and package build checks.
4. Commit the release metadata/docs and cleanup sweep together.
5. Tag, push, create the GitHub release, and upload the PyPI artifacts.

## Validation

- `pre-commit run --all-files`
- `invoke ci`
- `python -m build`
- `mkdocs build --strict`

## Rollback

If validation fails before publishing, revert the release metadata/docs commit and keep the previous release as latest. If publication succeeds but a release-blocking issue is found, publish a newer patch release.

## Decision Log

- Chosen: include the architecture cleanup sweep in the `0.9.1` release artifact set because the release commit already contains the finalized split and the docs need to describe the same operating surface.
- Rejected: splitting the cleanup sweep into a later patch release because that would leave the release note and the code surface out of sync.
