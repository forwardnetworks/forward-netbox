# Release Plan: 0.6.4

## Goal

Publish `0.6.4` with the optional routing and peering beta surface, including native `netbox-routing` BGP/OSPF objects, optional `netbox-peering-manager` sessions, routing diagnostics, and query-ID-aware built-in map execution.

## Constraints

- Keep routing and peering explicitly beta and disabled unless `enable_bgp_sync` is enabled.
- Keep NetBox 4.5.9 as the validated runtime and NetBox 4.5.x as the documented support target.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or private screenshots.
- Require local `invoke ci`, GitHub Actions success on `main`, and GitHub Actions success on the release tag before publishing artifacts.
- Publish immutable GitHub and PyPI artifacts from the tagged commit.

## Touched Surfaces

- Package metadata in `pyproject.toml` and `forward_netbox/__init__.py`
- Release compatibility and version-history docs
- Previously completed routing/peering implementation and diagnostic plans
- GitHub release and PyPI artifacts

## Approach

1. Bump package and plugin version metadata to `0.6.4`.
2. Update release compatibility tables and install examples.
3. Call out optional routing and peering import as beta in release notes.
4. Re-run the repo release gate and sensitive-content scan.
5. Commit, tag, push, verify GitHub CI on both branch and tag, then publish GitHub Release and PyPI artifacts.

## Validation

- Local `invoke ci` passed after the `0.6.4` version bump, including harness checks, sensitive-content pre-commit hook, Docker build/startup checks, Django checks, plugin tests, Playwright UI harness, docs build, and package build.
- Customer-specific sensitive-content scan passed for known live identifiers, credentials, and PyPI token patterns.
- `python -m twine check dist/forward_netbox-0.6.4*` passed for the locally built wheel and source archive.
- Pending GitHub Actions verification on `main` and `v0.6.4`.
- Pending PyPI upload verification.

## Rollback

If validation fails before publishing, revert the release metadata/docs commit and keep `0.6.3` as latest. If publication succeeds but a release-blocking issue is found, publish a newer patch release because GitHub release assets and PyPI files should remain immutable.

## Decision Log

- Chosen: `0.6.4` patch release because the optional routing and peering surface is feature-flagged beta and the rest of the default inventory path remains unchanged.
- Rejected: removing beta labeling from routing and peering because the implementation depends on optional external NetBox plugins and should be reviewed carefully before branch merge.
