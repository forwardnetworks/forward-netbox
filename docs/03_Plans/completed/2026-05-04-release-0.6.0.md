# Release Plan: 0.6.0

## Goal

Publish `0.6.0` with the native module import surface marked clearly as beta, plus the latest inventory normalization, module readiness, and module sync behavior.

## Constraints

- Keep NetBox 4.5.9 / 4.5.x compatibility wording aligned across package docs.
- Make the `dcim.module` path visibly beta in user docs and release notes.
- Do not commit customer identifiers, tenant names, network IDs, snapshot IDs, credentials, or live sample rows.
- Publish immutable PyPI/GitHub artifacts only after the full local gate passes.

## Touched Surfaces

- `pyproject.toml`
- `forward_netbox/__init__.py`
- `README.md`
- `docs/README.md`
- `docs/01_User_Guide/README.md`
- `docs/01_User_Guide/usage.md`
- `docs/02_Reference/built-in-nqe-maps.md`
- `docs/02_Reference/model-mapping-matrix.md`

## Approach

Bump package and plugin metadata to `0.6.0`, update compatibility/version tables, and state that native `dcim.module` import remains beta. Build the release from the validated main branch after the full CI gate and sensitive-content scan pass.

## Validation

- `invoke ci` passed after the `0.6.0` version bump, including harness checks, sensitive-content pre-commit hook, Docker build/startup checks, Django checks, plugin tests, Playwright UI harness, docs build, and package build.
- `python scripts/check_sensitive_content.py .gitignore docs forward_netbox tasks.py README.md pyproject.toml` passed.
- The local build produced `dist/forward_netbox-0.6.0.tar.gz` and `dist/forward_netbox-0.6.0-py3-none-any.whl`.
- Pending after push: GitHub Actions check for the release commit/tag.

## Rollback

If local validation fails, do not publish. If publication succeeds but a release-blocking issue is later found, cut a follow-up patch release because PyPI artifacts are immutable.

## Decision Log

- Chosen: `0.6.0` minor release because native module import adds a new NetBox model surface.
- Chosen: explicit beta labeling for `dcim.module` rather than hiding the feature, because operators need the low-friction workflow but should review staged module and module-bay changes carefully.
