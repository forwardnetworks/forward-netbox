# 1.3.6 Parent Coverage and Production Hardening

## Goal

Close the remaining production gaps needed for a 1.3.6 release by fixing
parent-child coverage for interface imports and finishing the last set of
operator-visible hardening items. The desired outcome is a sync path that:

- does not emit avoidable child-row failures when the parent row is missing
  from the same run,
- proves repeat-sync idempotence on real field data,
- keeps Forward API/NQE usage bounded and query-contract safe,
- exposes enough evidence in health and support-bundle surfaces to troubleshoot
  production runs without guesswork.

## Current Status

- Missing parent-device coverage is now reported once as structured dependency
  evidence instead of exploding into repeated row failures.
- The sync path skips only child rows blocked by that missing parent coverage,
  records the affected model and sample row identities, and exports that
  summary to support-bundle and health surfaces.
- The relevant validation gates are green: targeted tests, support-bundle
  checks, query-contract confirmation, and repo lint.

## Constraints

- Do not use retries, backoffs, or other workarounds to mask a broken data
  contract.
- Keep NQE as the source of truth for row shaping and keep sync-time column
  filters out of the execution path.
- Do not reintroduce per-value query execution or unbounded API/NQE fan-out.
- Do not commit customer identifiers, live snapshot IDs, network IDs,
  credentials, screenshots, or raw customer rows.
- Keep the 4.5.9 and 4.6.1 release lanes green.
- Preserve the current optional-plugin and branch-native architecture rather
  than introducing a one-off sync path.

## Approach

1. Fix the parent coverage contract for `dcim.interface`.
   - Verify that every interface row references a device that is present in the
     same sync run.
   - If the device row is missing, record one structured dependency issue that
     identifies the missing device and the affected interface rows.
   - Skip only the rows blocked by that missing parent coverage. Do not retry
     them and do not hide the mismatch behind a generic failure.
2. Extend the same parent-coverage logic to other child models that depend on
   a parent object already being present in the current run.
   - Priority order:
     - `dcim.macaddress`
     - `dcim.cable`
     - `ipam.ipaddress`
     - `ipam.fhrpgroup`
     - routing plugin models
     - module and inventory-item paths where parent identity is required
3. Align the query contract for the device/interface path.
   - Make sure the saved `dcim.device` and `dcim.interface` maps are both
     bounded, parameterized where appropriate, and compatible with the same
     shard/filter contract.
   - Keep the interface query from outpacing its parent device coverage.
4. Add explicit production evidence for this class of failure.
   - Surface missing-parent counts and names in the support bundle.
   - Surface the same summary in health and sync-issue views.
   - Keep the support artifact sufficient to diagnose a failed sync without
     needing a second repro run.
5. Prove repeat-sync idempotence on field data before release.
   - Run the revised sync against the approved dataset.
   - Confirm that a rerun does not reintroduce unexpected updates after the
     parent coverage fix lands.
   - Confirm that the interface path no longer fails when the parent device is
     present in the run.
6. Fold the remaining production-hardening checks into the same release gate.
   - Keep query-contract checks green.
   - Keep support-bundle evidence complete enough for troubleshooting.
   - Keep CI green on both supported NetBox versions.

## Rollback

- Revert the parent-coverage preflight and the related issue-reporting changes.
- Restore the previous query-contract behavior if the bounded map update causes
  a regression.
- If the release gate fails, keep 1.3.6 uncut and continue shipping the
  previous validated build.

## Decision Log

- Chosen: fail visibly on missing parent coverage instead of retrying child
  rows. This preserves the data contract and keeps production behavior
  explainable.
- Chosen: keep the device/interface fix in the same tranche as the remaining
  production-hardening items. The interface failure is one symptom of the same
  broader contract gap.
- Rejected: solving the problem with repeated sync attempts or row-level
  retries. That would hide the real mismatch between parent and child data.
