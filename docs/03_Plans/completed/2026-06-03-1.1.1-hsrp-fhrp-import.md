# Forward NetBox 1.1.1 HSRP FHRP Import

## Goal

Add optional HSRP group import for NetBox `ipam.FHRPGroup` using Forward's native
FHRP NQE fields, without adding excessive Forward API/NQE calls.

The same release also fixes the selected-device-tag scope regression for sites
and prefixes. Bundled site and prefix NQEs now accept device tag include/exclude
parameters so scoped imports do not collect broad site or prefix rows before
local pruning.

## Constraints

- Keep the feature optional by default.
- Use one paged Forward NQE for HSRP rows; do not run per-device,
  per-interface, per-group, or per-VIP Forward calls.
- Preserve the 1.1 source-level Forward API limiter and normal NQE pagination.
- Preserve the selected device tag scope for devices, sites, and prefixes.
- Keep tag-scoped site/prefix behavior source-side for bundled queries and
  compatible org query paths without breaking custom queries that do not declare
  tag parameters.
- Do not parse raw device configs in the 1.1.1 MVP.
- Do not commit customer identifiers, network IDs, snapshot IDs, credentials, or
  raw live rows.
- Keep NetBox-native FHRP objects as the target model surface.

## Approach

- Register optional model `ipam.fhrpgroup`.
- Add a single built-in HSRP NQE that emits NetBox-shaped rows:
  - `protocol`
  - `group_id`
  - `name`
  - `device`
  - `interface`
  - `vrf`
  - `address`
  - `state`
  - `priority`
  - `status`
- Use Forward native fields:
  - `*.ipv4.fhrp.hsrp.fhrpGroups`
  - `*.ipv6.fhrp.hsrp.fhrpGroups`
- Import into NetBox as:
  - one `ipam.FHRPGroup`
  - one `ipam.FHRPGroupAssignment` per device/interface/group row
  - one VIP `ipam.IPAddress` assigned to the FHRP group with role `hsrp`
- Use default assignment priority `100` because the Forward FHRP schema exposes
  state and group ID, but not HSRP priority.
- Keep Forward `MASTER`/`BACKUP`/`OTHER` state in the NQE row shape for future
  custom-field support, but do not write it to native NetBox fields in MVP
  because NetBox has no native assignment state field.
- Add source-side tag parameters to bundled site and prefix queries:
  - `device_tag_include_tags`
  - `device_tag_include_match`
  - `device_tag_exclude_tags`
- Keep prefix shard parameters in the same `@query` signatures so prefix
  sharding and selected tag scope can both reduce Forward NQE result volume.
- Push selected device tag parameters into tag-aware built-in query specs at
  fetch time. Query maps that do not expose these parameters are not sent
  incompatible parameter payloads.
- Store scoped site names and slugs in the fetch context so site rows with no
  device column can still be pruned consistently when local pruning is enabled.
- Treat an enabled tag scope that matches zero devices as an empty scope, not as
  a broad import.

## Hardening

- FHRP VIP import updates only an unassigned IP address or an IP address already
  assigned to the same FHRP group.
- If the VIP host already exists and belongs to another NetBox object, the FHRP
  row is skipped with an aggregated warning instead of reassigning ownership.
- Multiple devices/interfaces for the same protocol, group ID, VIP, and VRF
  share one NetBox FHRP group and one VIP.
- Deleting one participant keeps the shared group/VIP until the last assignment
  is removed.
- Same group/VIP identities in different VRFs remain separate.
- `ipam.fhrpgroup` remains adapter-required because it writes generic relations
  and must preserve row-level dependency/skip behavior before any faster apply
  engine can handle it.
- Site rows are retained only when their name or slug matches the scoped device
  locations collected from the tag-scope preview query.
- Prefix rows use source-side tag parameters because prefix rows do not carry a
  stable originating-device field after `select distinct`.
- Custom query maps remain compatible: local tag mode only injects tag
  parameters into specs that declare them.

## Rollback

Disable `ipam.fhrpgroup` in sync parameters or disable the `Forward HSRP Groups`
NQE map. Removing the feature code reverts to the prior 1.1 behavior because no
existing required model depends on HSRP import.

For tag-scoped sites and prefixes, rollback is to republish the prior site and
prefix NQEs or bind custom query maps without tag parameters. The plugin still
keeps local tag pruning for query maps that do not declare source-side tag
parameters.

## Decision Log

- HSRP is implemented before VRRP to match the immediate request and avoid
  surprise-importing another FHRP protocol in 1.1.1.
- Device-config parsing is rejected for MVP because Forward exposes native FHRP
  fields and config parsing would increase complexity without improving API/NQE
  behavior.
- Per-row Forward enrichment is rejected. Missing priority is defaulted to `100`
  instead of adding extra calls or config parsing.
- VIP conflict handling is strict across the project-standard ingestion bar:
  imported rows can update what they own or attach to unassigned native objects,
  but they must not silently steal ownership from another NetBox object.
- Site and prefix tag scope is handled in the bundled NQE layer because these
  models do not always carry a device key after the query projection. Fetching
  broad rows and pruning later would still create unnecessary Forward SaaS NQE
  and result-volume pressure.
- Zero matched devices under an enabled tag scope means zero in-scope rows. It
  must not silently fall back to a broad import.
