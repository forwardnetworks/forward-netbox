# Model Mapping Matrix

This matrix summarizes how the shipped Forward NQE maps populate each NetBox model.

Maps for `netbox_routing.*`, `netbox_peering_manager.*`, `netbox_cisco_aci.*`, and `netbox_dlm.*` require the matching optional plugin — install the **integrations** edition (`pip install forward-netbox[integrations]`, or a single extra like `[dlm]`) and enable the map. The **core** edition (`pip install forward-netbox`) ships only the NetBox-builtin models below. See [Editions](../01_User_Guide/README.md#editions-core-vs-integrations).

`Exact` means the Forward data maps cleanly to the NetBox object without a major semantic compromise. `Best-fit` means the plugin intentionally approximates the Forward concept to the closest useful NetBox model.

| NetBox Model | Built-In Query | Forward Source Shape | Mapping Style | Notes |
| --- | --- | --- | --- | --- |
| `dcim.site` | `Forward Locations` | `network.locations` filtered by collected device locations | Exact | Site names/slugs are shaped in NQE. |
| `dcim.manufacturer` | `Forward Device Vendors` | `device.platform.vendor` | Exact | One manufacturer per Forward vendor enum, with built-in names and slugs canonicalized in NQE. |
| `dcim.devicerole` | `Forward Device Types` | `device.platform.deviceType` | Best-fit | Forward device type is used as the NetBox device role. |
| `dcim.platform` | `Forward Platforms` | `device.platform.os` plus vendor | Exact | Platform slugs are shaped in NQE. APIC controllers stay on `APIC`, CIMC-managed hardware stays on `CIMC`, and ACI switches stay on `ACI`. |
| `dcim.devicetype` | `Forward Device Models` | `device.platform.model` plus vendor | Best-fit | Forward model is used as both model and part number. |
| `dcim.device` | `Forward Devices` | `network.devices` | Exact | Device lookup is by NetBox device name. |
| `dcim.virtualchassis` | `Forward Virtual Chassis` | None by default | Exact only when customized | The bundled map is conservative and emits no rows. Forward HA peer relationships such as vPC, MLAG, and active/standby clusters are separate control-plane relationships, not native NetBox virtual chassis membership. |
| `extras.taggeditem` | `Forward Device Feature Tags` | Forward structured protocol feature evidence | Best-fit | Adds NetBox tags to devices by exact device name; the default map tags BGP-enabled devices as `Prot_BGP`, and the disabled rules-aware variant can source tag policy from `netbox_feature_tag_rules`. |
| `dcim.interface` | `Forward Interfaces` | Interfaces under `device.interfaces` | Best-fit | NetBox interface type is derived from negotiated speed lookup values for physical Ethernet rows. Forward aggregate interfaces map to native NetBox LAG interfaces, and physical members attach through `Interface.lag` when Forward reports `ethernet.aggregateId`. A member can create a minimal native LAG placeholder when sharding applies the member before the aggregate row. MTU is preserved from Forward's normalized `interface.mtu` value. Forward switched access mode and trunk native VLAN are mapped to native NetBox `Interface.mode` and `Interface.untagged_vlan` when the site-scoped VLAN already exists; trunk tagged VLAN expansion remains intentionally deferred. |
| `dcim.cable` | `Forward Inferred Interface Cables` | `interface.links` resolved from Forward topology discovery | Best-fit | Creates cables only when the resolved neighbor device and port exactly match NetBox device/interface names. |
| `dcim.macaddress` | `Forward MAC Addresses` | `interface.ethernet.macAddress` | Exact | MACs are assigned to interfaces by exact interface name. |
| `dcim.inventoryitem` | `Forward Inventory Items` | `device.platform.components` | Best-fit | Forward hardware component part type is used as the inventory item role. Unknown part IDs and serial numbers remain blank rather than synthesized; application pseudo-parts are excluded. |
| `dcim.inventoryitem` | `Forward ACI APIC CIMC Inventory` | Parsed APIC custom command `moquery -c eqptCh -a all` joined to `CISCO_APIC_CONTROLLER_DETAIL` | Best-fit; optional | Requires Forward to collect the APIC custom command. The map emits one native NetBox inventory item named `CIMC` for each APIC controller with non-empty CIMC version, model, and serial evidence. |
| `dcim.module` | `Forward Modules` | `device.platform.components` filtered to line-card, supervisor, fabric-module, and routing-engine part types | Best-fit; beta | Bay-aware module rows can model chassis blades and similar replaceable hardware. Missing module bays must be imported first with native NetBox module-bay import; rows whose bays are missing are skipped to keep Branching merges deterministic, and matching generic inventory rows are cleaned up to avoid duplicate modeling. |
| `netbox_routing.bgppeer` | `Forward BGP Peers` | `networkInstances[].protocols[].bgp.neighbors` | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and the optional `netbox-routing` plugin. The query uses explicit local AS when present and can infer it from reciprocal Forward peer evidence or explicit internal-BGP peer AS; the adapter creates or reuses core NetBox ASNs, VRFs, peer IP addresses, BGP routers, scopes, and peers from Forward structured BGP neighbor state. |
| `netbox_routing.bgpaddressfamily` | `Forward BGP Address Families` | `device.bgpRib.afiSafis` joined to structured BGP neighbors | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and `netbox-routing`. The adapter creates native BGP address families on the matching router scope and normalizes Forward `L3VPN_*` names to the native `vpnv4-*` and `vpnv6-*` choices. |
| `netbox_routing.bgppeeraddressfamily` | `Forward BGP Peer Address Families` | `device.bgpRib.afiSafis[].neighbors` joined to structured BGP neighbors | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and `netbox-routing`. The adapter links native BGP peers to their observed AFI/SAFI rows and uses the same address-family normalization as `netbox_routing.bgpaddressfamily`. |
| `netbox_routing.ospfinstance` | `Forward OSPF Instances` | `networkInstances[].protocols[].ospf.areas[].neighbors` | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and `netbox-routing`. The query infers a unique process-level local router ID from reciprocal Forward OSPF neighbor evidence. Forward named process IDs are converted to deterministic numeric NetBox process IDs while preserving the original label in comments. |
| `netbox_routing.ospfarea` | `Forward OSPF Areas` | `networkInstances[].protocols[].ospf.areas` | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and `netbox-routing`. OSPF area type is mapped to native NetBox routing choices. |
| `netbox_routing.ospfinterface` | `Forward OSPF Interfaces` | `networkInstances[].protocols[].ospf.areas[].neighbors` | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync` and `netbox-routing`. The adapter binds OSPF instances and areas to exact native NetBox interfaces, including local rows whose specific neighbor lacks remote-peer inference when the process has unique router ID evidence elsewhere. |
| `netbox_peering_manager.peeringsession` | `Forward Peering Sessions` | `networkInstances[].protocols[].bgp.neighbors` | Best-fit; beta; feature-flagged | Requires `enable_bgp_sync`, `netbox-routing`, and `netbox-peering-manager`. The map links a peering-manager session to the native `netbox-routing` BGP peer and uses peer type as a simple relationship hint. |
| `ipam.vlan` | `Forward VLANs` | `device.networkInstances[].vlans` | Best-fit | VLANs are site-scoped using the device location. |
| `ipam.vrf` | `Forward VRFs` | non-default `device.networkInstances` | Best-fit | Route distinguisher is not supplied by Forward and remains null. |
| `ipam.prefix` | `Forward IPv4 Prefixes` | `networkInstances[].afts.ipv4Unicast.ipEntries` | Best-fit | Host-route receive/drop entries and clearly non-importable `0.0.0.0/8` or loopback route artifacts are filtered in NQE. |
| `ipam.prefix` | `Forward IPv6 Prefixes` | `networkInstances[].afts.ipv6Unicast.ipEntries` | Best-fit | Host-route receive/drop entries are filtered in NQE. |
| `ipam.ipaddress` | `Forward IP Addresses` | interface, subinterface, bridge, tunnel, and routed VLAN L3 addresses | Best-fit | Subinterface addresses are anchored to the parent interface name so NetBox interface lookups stay deterministic. Subnet network IDs and IPv4 broadcast addresses that NetBox cannot assign to interfaces are skipped rather than rewritten; global-table host IP collisions are canonicalized in NQE so the most specific surviving mask imports once; read-only diagnostics report filtered counts and full-run IP rows without imported parent prefixes. |
| `ipam.fhrpgroup` | `Forward HSRP Groups` | native Forward HSRP and VRRP group state on subinterfaces and routed VLANs | Best-fit; optional | Creates native NetBox FHRP groups, interface assignments, and VIP IP addresses from Forward structured HSRP and VRRP data. IPv4 VRRP maps to NetBox `vrrp2`, IPv6 VRRP maps to NetBox `vrrp3`, and VIP IP addresses use the native `vrrp` role. The adapter shares one group/VIP across multiple participating interfaces, separates identities by VRF, and skips rows rather than reassigning an existing IP address that belongs to another NetBox object. |
| `netbox_cisco_aci.acifabric` | `Forward ACI Fabrics` | ACI-mode devices and `CISCO_ACI_FABRIC_NODES` command output | Best-fit; optional plugin | Requires the optional `netbox-cisco-aci` plugin. The map emits a fabric container from the Forward location context without exposing raw command response text. |
| `netbox_cisco_aci.acipod` | `Forward ACI Pods` | Parsed `CISCO_ACI_FABRIC_NODES` command output | Best-fit; optional plugin | Requires the optional `netbox-cisco-aci` plugin. Pods are keyed by fabric plus pod ID and stay disabled until the operator enables the optional map. |
| `netbox_cisco_aci.acinode` | `Forward ACI Nodes` | Parsed `CISCO_ACI_FABRIC_NODES` command output | Best-fit; optional plugin | Requires the optional `netbox-cisco-aci` plugin. Nodes are keyed by pod plus node ID, preserve serial number and role, and link to native NetBox devices by exact node object name when the matching `dcim.device` row already exists. |
| `netbox_cisco_aci.acinode` | `Forward ACI APIC Nodes` | Parsed `CISCO_APIC_SWITCH` and `CISCO_APIC_CONTROLLER_DETAIL` command output | Best-fit; optional plugin | Requires the optional `netbox-cisco-aci` plugin. APIC node rows reuse the same plugin model as fabric nodes, keep APIC controllers and APIC-facing switches bounded to the parsed detail output, and do not expose raw responses. |
| `netbox_cisco_aci.acitenant` | `Forward ACI Tenants` | Parsed `moquery -c fvCtx` custom command output | Best-fit; optional plugin | Tenants are inferred from tenant-qualified APIC context rows and keyed under the fabric. |
| `netbox_cisco_aci.acivrf` | `Forward ACI VRFs` | Parsed `moquery -c fvCtx` custom command output | Best-fit; optional plugin | ACI VRF policy rows stay distinct from native `ipam.vrf` rows and use plugin defaults for policy flags that Forward does not expose. |
| `netbox_cisco_aci.acibridgedomain` | `Forward ACI Bridge Domains` | Parsed `moquery -c fvBD` custom command output | Best-fit; optional plugin | The adapter and row contract are present, and the built-in map now parses bounded BD identity from the APIC custom command rather than a broad VLAN/endpoint join. |
| `netbox_cisco_aci.aciappprofile` | `Forward ACI Application Profiles` | None by default | Contract-only; optional plugin | The built-in map emits no rows until application-profile identity is exposed or safely parsed. |
| `netbox_cisco_aci.aciendpointgroup` | `Forward ACI Endpoint Groups` | None by default | Contract-only; optional plugin | The built-in map emits no rows until EPG, app-profile, and bridge-domain identity can be proven repeatably. |
| `netbox_cisco_aci.acicontract` | `Forward ACI Contracts` | None by default | Contract-only; optional plugin | The adapter and row contract are present, but the built-in map emits no rows until zoning-rule parsing is bounded and repeat-sync safe. |
| `netbox_cisco_aci.acifilter` | `Forward ACI Filters` | Parsed `CISCO_ACI_ZONING_FILTER` command output | Best-fit; optional plugin | Filter rows are normalized without importing raw filter entries in this tranche. |
| `netbox_cisco_aci.acil3out` | `Forward ACI L3Outs` | Parsed `moquery -c l3extInstP` custom command output | Best-fit; optional plugin | The adapter and row contract are present, and the built-in map now parses bounded L3Out identity plus protocol ownership from the APIC custom command. |
| `netbox_cisco_aci.acistaticportbinding` | `Forward ACI Static Port Bindings` | None by default | Contract-only; optional plugin | The adapter can bind an EPG to an existing native interface, but the built-in map emits no rows until binding identity is safely parsed. |

## Important Caveats

- `dcim.virtualchassis` is intentionally no-op by default. If you need this model, use a custom NQE map that emits true NetBox virtual chassis membership rows with unique `vc_position` values.
- Optional plugin capability reporting is shared across routing, peering, and Cisco ACI. The architecture audit, sync health page, and support bundle all show the same installed/available/version status for those optional surfaces.
- The plugin intentionally keeps NetBox-ready shaping in NQE where possible. Python adapters apply rows and enforce object lookups; they should not silently normalize meaning after the query runs.
- Manufacturer-bearing built-in queries intentionally canonicalize vendor names and slugs in NQE through the shared `netbox_utilities` module. If your NetBox already uses different curated manufacturer rows with the default maps, copy the query set and update `manufacturer_name_overrides` there before syncing.
- Disabled alias-aware variants for `dcim.devicetype` and `dcim.device` can map Forward model strings to NetBox Device Type Library model/slug values through the `netbox_device_type_aliases` Forward data file. That data file can also carry manufacturer override rows for the alias-aware maps. Keep the default maps enabled unless the selected Forward snapshot exposes the data file value.
- The disabled feature-tag rules variant can map Forward structured features to NetBox tags through the `netbox_feature_tag_rules` Forward data file. Keep the default feature-tag map enabled unless the selected Forward snapshot exposes that data file value.
- Optional routing sync is enabled unless `PLUGINS_CONFIG["forward_netbox"]["enable_bgp_sync"]` is false. BGP route-table import and config-derived policy objects such as route maps, prefix lists, and communities are intentionally out of scope for the normalized beta path. When optional routing maps are enabled, an internal read-only diagnostic query reports BGP neighbors skipped because Forward did not expose or imply a usable local AS, unsupported BGP address-family values, and OSPF rows skipped because Forward did not expose unique router ID inference required for native OSPF objects.
- Interface, cable, and IP assignment remain intentionally strict: if the built-in queries drift from exact interface names, the sync should record issues rather than guessing. Interface VLAN assignment only uses already-imported site-scoped VLANs; it logs a warning instead of creating VLANs from interface rows.
- FHRP import is optional because not every deployment wants Forward to own NetBox FHRP state. Enable `ipam.fhrpgroup` only after interface and IPAM imports are stable for the source dataset.
- Cisco ACI plugin import is optional because it depends on an external NetBox plugin and parsed Forward command output. Enable the ACI fabric, pod, node, tenant, VRF, and filter maps after native `dcim.device` import is stable. Bridge domains, app profiles, EPGs, contracts, L3Outs, and static bindings are present as disabled contract-only maps until their NQE field contracts and repeat-sync behavior are proven.
